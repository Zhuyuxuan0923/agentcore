LLM 内部原理 Day 2 — Transformer + Attention：LLM 怎么"理解"上下文？
================================================================
日期：2026-06-13
模块：LLM 内部原理速通（Day 2/3）


零、今天学什么
===============

昨天你知道了 LLM 用 Tokenization 把文字变成数字 ID。
但这些数字 ID 进入模型之后，到底发生了什么？

今天回答一个问题：**LLM 是如何"读懂"一句话里每个词之间的关系的？**

答案是 Attention（注意力机制）。这是整个现代 AI 革命的核心发明。
2017 年一篇叫 "Attention Is All You Need" 的论文提出了它，
ChatGPT 的 "T"（Transformer）就是这个架构的名字。

读完今天这篇，你会理解：
- "LLM 能理解上下文"不是魔法，而是一个叫"注意力"的机制
- 为什么上下文窗口是平方级成本（O(n²)），不是你想象中"多加几个字无所谓"
- Prompt 里信息的顺序为什么影响结果（"放前面"和"放后面"效果不同）

⚠️ 今天不需要学数学公式。目标是建立心智模型，不是手算矩阵。


一、Transformer 之前：旧方法有什么致命缺陷？
==============================================

▶ RNN（循环神经网络）——"用一只眼睛从左到右读"

在 Transformer 出现之前，AI 读文字的方式叫 RNN（Recurrent Neural Network）。
它读文字的方式就像你用一只眼睛通过一道细缝看一句话——

  输入："我今天下午要去图书馆借一本书"
  RNN 读法：
  "我" → "今天" → "下午" → "要" → "去" → "图书馆" → "借" → "一本" → "书"

  每读一个新字，它都必须"记住"之前读到的所有内容。
  因为只有一个狭窄的通道从左到右传送信息。

▶ RNN 的三个致命缺陷

  缺陷 1：遗忘
    读到"书"的时候，它可能已经忘了最前面的"我"。
    就像你一口气读了一篇 5000 字的文章，读到最后一个自然段时，
    第一段的细节早就忘了。

  缺陷 2：串行处理 —— 不能加速
    RNN 必须读完第 1 个字才能读第 2 个，读完第 2 个才能读第 3 个。
    无法并行。想象一本 500 页的书，你只能一页一页翻——训练特别慢。

  缺陷 3：远距离关系完全抓不到
    "小明今天很晚才回家，因为他最喜欢的老师给他补习了数学。他___很累。"
    RNN 读到空格处，很难把"他"和 20 个字之前的"小明"关联起来。
    因为它只有一个从左到右的"传送带"，信息在路上已经衰减了。

▶ 一句话总结：RNN 像用吸管喝一大碗汤——又慢又喝不着底。


二、什么是 Self-Attention（自注意力）？
========================================

▶ 生活类比：用荧光笔读一篇论文

想象你在读一篇 10 页的学术论文，准备回答"作者的核心观点是什么"。

你的眼睛不是一行一行按顺序扫的——
而是先看标题，再看摘要，然后跳到结论段，划几处重点，偶尔回看引言
确认某个概念的定义。这个过程你其实在潜意识中做了三件事：

  1. 对每个词打分："这个词对理解核心观点重要吗？"
  2. 在不同位置之间建立联系："第 3 页这个实验描述和第 7 页这个结论有关"
  3. 综合所有线索，形成一个全局理解

Self-Attention 做的就是这个——让每个词能够**同时看到句子中的所有其他词**，
并决定"我应该关注谁"。

▶ 和 RNN 的关键区别

  RNN 读法（串行，一人传话）：
  我 → 今 → 天 → 下 → 午 → 要 → 去 → 图 → 书 → 馆 → 借 → 书
  每个字只和紧挨着的上一个字有直接关系。

  Transformer 读法（并行，全部照面）：
              ┌─────┐
        ┌─────┤  我  ├─────┐
        │     └─────┘     │
    ┌───┴──┐         ┌───┴──┐
    │ 今天  │         │  要   │  ...  ← 每个词可以同时"看"到所有
    └───┬──┘         └───┬──┘      其他词，计算注意力分数
        │     ┌─────┐     │
        └─────┤ 去   ├─────┘
              └─────┘

  所有词在同一个时间点被处理，每个词对所有其他词都有一个"注意力分数"。


三、Attention 怎么算？（只用类比，不用公式）
=============================================

每个 token 进入 Attention 层后，被赋予三个向量（数字列表）：

  Q（Query，查询）：  "我想找谁？"  ——这个词需要什么样的信息
  K（Key，键）：      "我是谁？"    ——这个词能提供什么信息
  V（Value，值）：    "我有什么？"  ——这个词的实际内容

▶ 生活类比：图书馆查资料

  你要写一篇关于"唐朝诗歌"的论文。

  Q = 你的检索需求："我需要关于唐朝诗歌的段落"
  K = 每本书的关键词标签：这本书在目录后面的标签写了"唐诗""李白""格律诗"
  V = 每本书的实际内容：那本书里的具体文字

  查找过程：
  1. 用你的 Q 去匹配所有书的 K——Q 和 K 越匹配，说明这本书越值得看
  2. 算出每本书的"匹配度分数"（数学上是 Q·K 的点积，你不用管这个）
  3. 用这个分数做权重，把所有书的 V（实际内容）加权平均
  4. 结果：你获得了"结合了最相关信息的综合理解"

▶ 回到 LLM 场景

  输入："小明把苹果给了小红，因为他___很饿。"
  LLM 要预测空格处是"觉得"还是"真的"还是"不"。

  Attention 会做什么？
  "他"的 Q 和所有词的 K 做匹配，发现"小明"的 K 和"他"的 Q 最匹配。
  所以"小明"的 V（语义信息）被高权重地传给"他"这个位置。
  LLM 于是知道"他"指的是"小明"，然后推理出空格应该是"觉得"——“小明觉得饿”合理。

▶ 同一个句子中 Attention 权重的实际表现

  "小明把苹果给了小红，因为他很饿。"

  "他"这个位置的 attention 权重分布：
  ┌────────┬──────────────────────────────────────────────────────────┐
  │ 小明   │ ████████████████████████████████████████████  高 (0.45) │
  │ 把     │ ██                                            低 (0.03) │
  │ 苹果   │ ████                                          低 (0.08) │
  │ 给了   │ ███                                           低 (0.05) │
  │ 小红   │ ████████                                      中 (0.15) │
  │ 因为   │ ██████                                        中 (0.12) │
  │ 他     │ ██                                    ← 自己 (0.02)     │
  │ 很     │ █████                                         中 (0.07) │
  │ 饿     │ ███                                           低 (0.03) │
  └────────┴──────────────────────────────────────────────────────────┘

  "他"把最高注意力给了"小明"（因为指代关系）和"小红"（给了谁），
  而把很低注意力给了"把""饿"这类功能词。

  这不就是人类阅读时的"划重点"吗？


四、Multi-Head Attention（多头注意力）
=====================================

▶ 一个头不够，要多个头

上面讲的只是一个"注意力头"——这个头发现了"他"指代"小明"（代词消解）。

但如果只有一个头，模型只能捕捉一种关系。
实际上，语言里有多种关系同时存在：

  "小明上周在北京花了三百块买了两本关于人工智能的书。"

  这短短一句话里，我们需要知道：
  - 谁买？（施事者）        → "小明"和"买"的关系
  - 买什么？（宾语）        → "买"和"书"的关系
  - 几本？（数量）          → "两本"和"书"的关系
  - 多少钱？（价格）        → "三百块"和"买"的关系
  - 在哪？（地点）          → "北京"和"买"的关系
  - 什么时候？（时间）      → "上周"和"买"的关系
  - 什么主题？（修饰）      → "人工智能"和"书"的关系

一个注意力头只能关注一类关系，所以需要多个头并行工作。

▶ Multi-Head = 多个"专家"同时阅读同一句话

  Head 1：专门抓"谁-做了什么"（主语-动词关系）
  Head 2：专门抓"修饰什么"（形容词/定语关系）
  Head 3：专门抓"指代关系"（代词消解）
  Head 4：专门抓"位置/时间"（地点/时间状语）
  ...（GPT-3 每个层有 96 个头）

每个头产生自己的注意力权重，最后把所有头的结果拼接在一起。
这就像一篇文章同时被 96 个专家审阅，每人从一个角度分析，最后综合意见。

六、完整的 Transformer 架构（一张图看完）
=========================================

你不需要理解每一层在做什么。知道整体结构就够了。

  ┌─────────────────────────────────────────────────────┐
  │                                                   │
  │         输入：一串 Token ID                         │
  │            │                                       │
  │            ▼                                       │
  │    ┌──────────────┐                                │
  │    │  Embedding   │  把 Token ID 转成向量            │
  │    │  + 位置编码   │  + 标明每个词的位置               │
  │    └──────────────┘                                │
  │            │                                       │
  │            ▼                                       │
  │    ╔══════════════╗                                │
  │    ║ 重复 N 次    ║  ← GPT-3: 96 层               │
  │    ║              ║    每层有：                      │
  │    ║  ┌────────────────┐                           │
  │    ║  │ Multi-Head     │  ★ 核心：让每个词看到       │
  │    ║  │ Self-Attention │    所有其他词               │
  │    ║  └────────────────┘                           │
  │    ║         │                                     │
  │    ║         ▼                                     │
  │    ║  ┌────────────────┐                           │
  │    ║  │  Feed-Forward  │  把注意力结果做非线        │
  │    ║  │  Network       │  性变换（增强表达）         │
  │    ║  └────────────────┘                           │
  │    ╚══════════════════╝                            │
  │            │                                       │
  │            ▼                                       │
  │    ┌──────────────┐                                │
  │    │ 输出层       │  输出每个可能 token 的概率       │
  │    │ (Softmax)    │  概率最高的 token 就是下一个字    │
  │    └──────────────┘                                │
  │            │                                       │
  │            ▼                                       │
  │    输出的下一个 Token                                │
  │                                                   │
  └─────────────────────────────────────────────────────┘

  流程：
  1. Token ID 进来 → Embedding 变成向量
  2. 加上位置编码（让模型知道"哪个在第1位、哪个在第2位"）
  3. 过 N 层 Transformer Block，每层有 Attention + Feed-Forward
  4. 最后一层输出：对每个可能的 token 计算概率
  5. 选概率最高的 token → 这就是 LLM "生成"的下一个字

  Week 1 你已经用过了——temperature 参数控制的就是第 5 步"选 token"时的随机程度。


七、为什么上下文窗口是瓶颈？（和你正在做的 RAG 直接相关）
==========================================================

现在你能理解 RAG 为什么是刚需了。

▶ Attention 的计算成本是平方级的

  每个 token 都要和所有其他 token 算注意力分数。

  10 个 token → 10×10 = 100 次计算
  100 个 token → 100×100 = 10,000 次计算
  1000 个 token → 1,000,000 次计算
  128,000 个 token → 16,384,000,000 次计算

  而且每一层都要重复！GPT-3 有 96 层。

  这就是为什么：
  1. 上下文窗口不是免费的——越长越贵（平方增长）
  2. LLM 推理需要大量 GPU 显存——显存随上下文长度平方增长
  3. 你不能把 100 本书直接塞进 prompt 让 LLM 读——太贵了，而且不现实

▶ 这也是为什么 RAG 的设计是"只检索相关片段"而不是"全量塞进去"

  ❌ 把 500 篇文档全塞进 prompt → 平方成本爆炸
  ✅ 只检索 Top-5 最相关的片段塞进去 → 上下文短，成本可控

▶ "Lost in the Middle" 现象

  研究表明，LLM 对 prompt 中间部分的信息关注度偏低。
  最开头和最末尾的内容更容易被注意，中间的内容容易被"忽略"。

  这对你的 prompt 设计意味着：
  - 关键指令放在 system prompt 开头或用户消息末尾
  - 不要把最重要的约束藏在长 prompt 的中间

  这也是 Week 2 Day 6 你测出来 negative_examples 放在前面效果更好的原因之一。


八、今天最重要的 4 个认知
==========================

1. LLM 不是"从左到右"理解文字的。Self-Attention 让每个 token 能同时看到
   所有其他 token，然后决定关注谁、忽略谁——就像你用荧光笔读文章。

2. Query/Key/Value 是同一个东西换了三个角色：
   我要找什么（Q）、我是谁（K）、我有什么（V）。
   不需要记公式，记住图书馆查资料的类比就够了。

3. Attention 是平方成本（O(n²)）。
   这就是为什么上下文不是越大越好、RAG 是刚需、prompt 要精简。
   这是你整个 AI Agent 技术选型的基础。

4. 位置信息（一个词排第几位）是额外加进去的。
   模型本身不知道顺序——"狗咬人"和"人咬狗"如果不加位置编码，
   模型会觉得是同一个意思。

十、补充知识：CNN 和 RNN 的现状
===============================

前文讲了 RNN 因为"串行处理 + 遗忘"被 Transformer 取代。
但你可能会问：CNN 是什么？它是不是也跟着 RNN 一起淘汰了？

▶ CNN 是什么？

CNN（卷积神经网络）= 用一个小窗口在图片上滑动，逐块扫描。

生活类比：你在一张照片上找猫。你不是一眼就看完整张照片——
目光是一个小区域一个小区域扫过去，检查"这里有耳朵吗？""这里有胡须吗？"
每扫一小块就做个判断，最后拼出结论。

CNN 就是这个"滑动窗口扫描"的数学版本。它天然适合图像——
因为图像里相邻像素有关联（一只猫的耳朵占了 50×50 像素的区域），
不需要像 Transformer 那样让每个像素去注意所有其他像素。

▶ 现在谁在用谁？

  ┌─────────────┬──────────────────────┬──────────────────────────────┐
  │   架构       │       主战场          │           现状               │
  ├─────────────┼──────────────────────┼──────────────────────────────┤
  │ CNN         │ 图像识别、目标检测、   │ 活得很好——ResNet/ConvNeXt    │
  │             │ 自动驾驶视觉系统       │ 还是视觉标配                  │
  ├─────────────┼──────────────────────┼──────────────────────────────┤
  │ RNN         │ 曾经的文字/语音处理    │ 基本退役，被 Transformer 取代  │
  ├─────────────┼──────────────────────┼──────────────────────────────┤
  │ Transformer │ NLP（全部）、图像（ViT │ 统治地位，正向图像领域扩展     │
  │             │ 挑战 CNN）、多模态     │                              │
  └─────────────┴──────────────────────┴──────────────────────────────┘

▶ 为什么 RNN 凉了但 CNN 没凉？

RNN 凉的原因就是本课第一节讲的：串行慢、忘开头。
Transformer 的 Self-Attention 在 RNN 的主场（文字处理）全方面碾压。

CNN 没凉因为图像和文字的结构不同：
  - 图像有空间局部性——相邻像素天然相关，CNN 的滑动窗口刚好匹配
  - 图像里大部分远距离关系不重要——左上角和右下角不需要互相注意
  - CNN 计算量远小于 Transformer，手机上也能跑

不过 ViT（Vision Transformer）正把 Transformer 引入图像领域——
把图切成小 patch 当 "token" 处理。未来可能是 CNN + Transformer 混合的天下。

▶ 对你的意义

作为 AI Agent 工程师，你最常打交道的是 Transformer（LLM 的核心架构）。
但如果你以后做多模态 Agent（比如让 AI 分析监控视频帧），
就需要同时理解 CNN（图像编码）和 Transformer（文本推理）。

Week 9 的多模态 RAG 会用到这个知识点，现在有个印象就行。
