LLM 内部原理 Day 1 — Tokenization：LLM 怎么"读"文字？
=======================================================
日期：2026-06-13
模块：LLM 内部原理速通（Day 1/3）


零、今天学什么
===============

你给 LLM 发了一句 "请帮我写一首关于春天的诗"。
LLM 看到的不是这 12 个字，而是一串数字：`[64790, 102947, 83594, 38372, 100167, 253, 109479, 245, 110148, 100]`

这个过程叫 Tokenization（分词/标记化）——把人类文字翻译成 LLM 能理解的数字 ID。

今天你不需要写代码，但你需要在概念上理解 token 是什么。因为：
- 看懂 OpenAI 账单上的 "1000 tokens" 到底是什么
- 理解为什么中文 prompt 比英文贵
- 排查 "输出的 JSON 被截断" 这类 bug 时，知道去哪里找原因


一、什么是 Token？
==================

▶ 最直白的定义

Token = LLM 的"最小阅读单位"。

  人类读文字：以"字"或"词"为单位    "我 / 爱 / 写 / 代码"
  LLM 读文字：以"Token"为单位       "我 / 爱 / 写 / 代 / 码"

对中文来说，一个 token 大约等于一个字（但不完全是）。
对英文来说，一个 token 大约等于 3/4 个单词或 4 个字符。

▶ 生活类比：罐头厂的切块机

  一家罐头厂，进料口只能接受 3cm³ 的小方块。
  不管来的是土豆还是牛肉，都得先切成标准小块才能送进机器。

  LLM 也一样——不管来什么语言的文字，都得先切成标准 Token，
  因为模型的"进料口"只认识数字 ID。

▶ 用 OpenAI 官方工具直观感受

打开浏览器访问：https://platform.openai.com/tokenizer
在里面输入中文和英文对比一下：

  输入："人工智能"         → 4 tokens（每个人工一个 = "人""工""智""能" 各1个）
  输入："Artificial Intelligence" → 2 tokens（"Artificial" 1个, "Intelligence" 1个）

这就是第一个重要发现：**同样的意思，中文 token 数通常是英文的 1.5-3 倍**。
意味着用中文调 API 天然更贵——因为你花的 token 更多。


二、Token 怎么来的？（BPE 分词算法）
====================================

LLM 不会凭空知道"哪些字符应该组成一个 token"。
它用的是一种叫 BPE（Byte Pair Encoding，字节对编码）的算法。
你不用理解算法细节，但需要知道它的核心逻辑：

▶ 一句话概括 BPE

  "统计哪些字符组合出现频率最高 → 把它们合并成一个 token → 重复"

▶ 图解过程（以英文为例）

  初始状态（每个字母是一个 token）：
  l o w _ l o w _ l o w e r _ l o w e s t

  第 1 轮：发现 "l"+"o" 出现次数最多 → 合并成 "lo"
  lo w _ lo w _ lo w e r _ lo w e s t

  第 2 轮：发现 "lo"+"w" 出现次数最多 → 合并成 "low"
  low _ low _ low e r _ low e s t

  第 3 轮：出现 "_low" 频率高 → 合并成 "_low"
  _low _low _lower _lowest

  最终词表里就有了：_low, er, est（不用每次重新拼）

▶ 为什么用 BPE？核心优势

  1. 它自动发现了"经常一起出现的几个字母"→节约 token 数
  2. 遇到从没见过的词（比如 "lowestation"），它能用已知子词拼出来：low + est + ation
     → 不会因为不认识就"卡住"

▶ 中文的 BPE 类似，但切分单位是"字"而不是"字母"

  常见中文 token：一个字、两个字的词、甚至三个字的常见搭配
  "人工智能技术" → ["人工", "智能", "技术"]  或  ["人", "工", "智能", "技术"]
  具体怎么切取决于训练数据和词表大小


三、这对你意味着什么？（直接影响开发的 5 件事）
===============================================

▶ 影响 1：上下文窗口（Context Window）

GPT-4o 上下文窗口 = 128,000 tokens。
这里的 128K 是 token 数，不是中文字数，也不是英文单词数。

  1000 个中文字 ≈ 1200-1500 tokens
  1000 个英文单词 ≈ 750-1000 tokens

所以：同一篇中文文档塞进 prompt，消耗的 token 配额比英文多 1.5 倍左右。
这就是 RAG 必须做分块（chunking）的根本原因之一——不切碎，上下文根本装不下几篇文档。

▶ 影响 2：计费

OpenAI 的定价按 token 计：

  GPT-4o 输入：$2.50 / 1M tokens，输出：$10.00 / 1M tokens

一个典型的 Agent 对话：
  输入 2000 tokens + 输出 500 tokens = 2500 tokens
  成本 ≈ (2000 × 0.0000025) + (500 × 0.00001) = $0.005 + $0.005 = $0.01

中文对话 ×3，大概 $0.03/次。1000 次 = $30。所以开发时注意用中文测试成本。

▶ 影响 3：模型选择

不同模型的 tokenizer 不同！同样的文字，GPT-4o 可能切成 100 tokens，DeepSeek 切出来 120。

用 Jinja2 模板做多 provider 切换时（你 Week 2 Day 2 做的），
要注意不同模型的 token 限制不同：
  GPT-4o: 128K
  Claude 3.5 Sonnet: 200K
  DeepSeek V3: 128K
  Gemini 1.5 Pro: 1M

▶ 影响 4：输出截断

如果你调用 API 时设置了 max_tokens=256，LLM 生成了 "{"name": "张三", "a"——
最后一个 "a" 后面应该还有 "ge": 24}"，但它到了 256 个 token 就被掐断了。

你的 JSON 解析器会直接报错。这就是 Week 2 Day 3 你做的 "五行 JSON 抢救" 要解决的问题之一。

▶ 影响 5：Prompt 设计

你在设计 System Prompt 时，以下几个因素都会吃掉 token 配额：
  - system prompt 本身
  - few-shot 示例（每个示例的输入+输出都算）
  - 对话历史（多轮对话每一轮都占 token）
  - RAG 检索到的文档片段
  - 加上 max_tokens 输出限额

所以好的 Agent 设计不只是"写得准"，还得"省 token"——
少用啰嗦话术，多轮对话超长时做摘要压缩。


四、三个关键数据（记住就行）
=============================

| 知识点 | 数值 |
|--------|------|
| 中文 1 个字 ≈ 多少 token | 1.2 - 1.5 tokens |
| 英文 1 个单词 ≈ 多少 token | 1 - 1.3 tokens |
| 1000 tokens 能读多少页中文 | 约 0.5-0.75 页（A4 五号字） |
| GPT-4o 上下文上限 | 128K tokens ≈ 8-10 万中文字 ≈ 一本中篇小说 |
| 输出 token 比输入 token 贵多少 | 输出约贵 4 倍（GPT-4o 定价） |
