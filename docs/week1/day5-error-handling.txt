今天学了什么

1. 理解了异常（Exception）的本质——不是代码写错了，是"外部世界不配合"。API 调用会因网络、限流、Key 错误等原因失败，不处理就崩
2. 掌握了 try/except 的正确写法——从具体异常（AuthenticationError）到通用异常（APIError），越具体越靠前
3. 学会了用 tenacity 库实现指数退避重试：wait_exponential 控制间隔、stop_after_attempt 控制次数、retry_if_exception 自定义判断条件
4. 发现了一个重要的"坑"：AuthenticationError 继承自 APIStatusError！如果只用 retry_if_exception_type(APIStatusError)，401 错误也会被重试——纯浪费
5. 学会了 Stream 中断恢复的思路——累积已收内容、重连时从断点继续
6. 写了 8 个单元测试，用 Mock 模拟各种异常场景，验证重试逻辑的正确性


核心概念

| 概念 | 一句话解释 | 类比 |
|------|-----------|------|
| 异常（Exception） | 程序运行时发生的"意外"，不是代码写错了 | 开车去上班，路被封了——不是你不会开车，是外部条件变了 |
| try/except | 给可能出错的代码包一层"安全气囊" | 汽车的安全气囊——平时不碍事，撞车时救命 |
| 重试（Retry） | 失败了再试一次，但不是无限试 | 电话打不通→过一会儿再拨，不是连续拨 100 次 |
| 指数退避（Exponential Backoff） | 每次重试等更久：1s→2s→4s→8s... | 敲门→等1秒→敲重一点→等2秒→再敲→等4秒 |
| 临时性错误 vs 永久性错误 | 网络抖动是临时的（值得重试），Key 错了是永久的（重试没用） | 临时关门（等下会开）vs 你拿错钥匙了（等多久都打不开） |
| Rate Limit（限流） | API 说"你太快了"，返回 HTTP 429 | 奶茶店一次只能做 5 杯，第 6 杯让你等 3 分钟 |
| tenacity | Python 最流行的重试库，把重试逻辑标准化 | 自动重拨电话机——设置好"最多打几次、间隔多久"，不用手动拨 |
| wait_exponential | 指数增长等待时间：multiplier × 2^attempt | 第一次等 1 分钟，第二次等 2 分钟，第三次等 4 分钟... |
| retry_if_exception | 用自定义函数判断"该重试吗"，比按类型匹配更精确 | 门卫不只看"来的是谁"，还看"今天让不让进" |
| Mock（模拟） | 测试时假装 API 返回某种错误，不用真的拔网线 | 地震模拟振动台——测试房子抗震不用等真地震 |
| pytest | Python 最流行的测试框架，test_ 开头的函数会自动被发现和运行 | 自动体检仪——把所有器官（函数）测一遍，告诉你哪个有问题 |
| Decorator（装饰器） | 给函数外包一层逻辑，不修改函数本身 | 在门外加了个门卫——你进去之前先过安检（重试逻辑） |

关键代码结构

```python
# ① 重试判断逻辑——比 retry_if_exception_type 更精确
def _should_retry(exc: BaseException) -> bool:
    # 永久性错误：不重试
    if isinstance(exc, (AuthenticationError, BadRequestError)):
        return False
    # 临时性错误：重试
    if isinstance(exc, (APIConnectionError, RateLimitError, APIStatusError)):
        return True
    # 不认识的：保守不重试
    return False

# ② 用 tenacity 构建重试装饰器
@retry(
    wait=wait_exponential(multiplier=1, min=1, max=30),
    stop=stop_after_attempt(max_retries + 1),
    retry=retry_if_exception(_should_retry),
    before_sleep=before_sleep_log(logger, logging.WARNING),
    reraise=True,
)

# ③ 更新后的 LLMClient 使用方式
client = LLMClient(provider="deepseek", max_retries=3)  # 自动重试
reply = client.chat("你好")  # 如果网络抖了，自动等 1s/2s/4s 重试
```


踩过的坑

1. 问题：测试里构造 APIConnectionError 时报 TypeError: missing required argument 'request'
   原因：新版 openai SDK（≥2.0）的异常类强制要求传 request 参数
   解决：用 httpx.Request("POST", "https://fake.api.example.com/...") 造一个假 request 对象传入

2. 问题：AuthenticationError 被 @retry 重试了，但按设计它不应该重试
   原因：AuthenticationError 继承自 APIStatusError（HTTP 401 = HTTP status error）。用 retry_if_exception_type(APIStatusError) 会覆盖所有 HTTP 错误，包括 401
   解决：改用 retry_if_exception(_should_retry) 自定义判断函数，显式排除 AuthenticationError 和 BadRequestError

3. 问题：before_sleep_log(sys.stdout, ...) 在测试中报 AttributeError: 'EncodedFile' object has no attribute 'log'
   原因：sys.stdout 不是 Python 的 logging.Logger 对象，只是一个文件流。tenacity 的 before_sleep_log 需要真正的 logger
   解决：用 logging.getLogger("xxx") 创建标准 logger，传给 before_sleep_log

4. 问题：ruff 报 E402 "Module level import not at top of file"
   原因：import 语句写在了函数定义之后，Python 规范要求所有 import 在文件顶部
   解决：把所有 import 移到文件最开头


发现的重要认知

1. "该不该重试"不是看异常类型，是看异常是否"临时性"。401 是永久性的（Key 错了），429 是临时性的（被限流），500 也是临时性的（服务器可能恢复）
2. tenacity 的 retry_if_exception_type 有"子类陷阱"——AuthenticationError 是 APIStatusError 的子类，按类型匹配会误伤。生产代码应该用 retry_if_exception 加自定义判断函数
3. 指数退避不只是"等久一点"，更是"防止雪崩"。100 个请求同时失败、同时等 1 秒、同时重试，服务器又被打爆——这就是"惊群效应"。加上 jitter（随机偏移）可以避免
4. Stream 中断恢复的本质是"状态保存"——你必须在每次收到 chunk 时把内容存下来，这样中断了才知道"已经收到哪了"。这和事务日志是同一个思想
5. Mock 测试让验证重试逻辑变得可重复——不用真的拔网线、等超时。好的测试是"可预测的假象"，跑 100 次结果一样
