Week 2 Day 1 — Prompt 工程理论学习日志
============================================
日期：2026-06-11
主题：精读 Prompt Engineering Guide，整理 10 条核心原则


一、什么是 Prompt Engineering
==============================

Prompt Engineering = 给 LLM 这个"超级实习生"写任务说明书。

LLM 的本质：通过海量文本训练，学会了"给定上文，预测下一个 token 的概率"。
它不是在"思考"，而是在做概率计算。模糊的 prompt 对应宽的概率分布——什么都可能出来。
精确的 prompt 把概率分布收窄到正确区域——出来的东西就靠谱。

核心认知：LLM 不会反问、不会主动澄清。你说的越模糊，它猜的越随机。


二、10 条核心 Prompt 工程原则
==============================

原则 1：把模型当"失忆的新员工"
--------------------------------
模型不知道你的习惯、场景、偏好。上下文要主动提供，不能假设它"应该知道"。
好 prompt = "你是一个有 10 年经验的 Python 后端工程师，请优化以下代码，目标是可读性和性能"
坏 prompt = "优化这段代码"

原则 2：好的 System Prompt 包含五个要素 ★
------------------------------------------
五要素：角色（Role）、上下文（Context）、任务（Task）、约束（Constraints）、输出格式（Format）
这五个要素缺一个，模型就会在那个维度自己"脑补"。
脑补 = 概率分布变宽 = 结果不可控。
这条是整个 Prompt Engineering 的骨架，后面 Week 2 Day 6 设计自己的 Agent System Prompt 时就用这个模板。

原则 3：Few-Shot 比 Zero-Shot 稳定
------------------------------------
Zero-Shot = 不给例子直接问，Few-Shot = 给 2-5 个输入→输出的范例。
例子相当于给模型一个"样本分布"，它直接模仿就行，不需要猜你要什么风格。
适用场景：输出格式特殊、风格难以用文字描述、任务边界模糊。

原则 4：让模型"先想再说"（Chain of Thought）
----------------------------------------------
不要直接要答案，让模型先输出推理过程、再给结论。
因为模型是一个 token 一个 token 地生成的，推理过程本身就是用前面生成的 token
来约束后面 token 的概率分布。相当于"自己给自己更精确的 prompt"。
写法：把"直接给出答案"换成"请先分析问题，列出已知条件，再逐步求解"。

原则 5：告诉模型"不能做什么"和"能做什么"同样重要
--------------------------------------------------
负面约束消除掉模型"犯错误"的方向。每加一条约束，概率分布就窄一圈。
好 prompt = "推荐 3 本书，必须是中文在售实体书，价格不超过 80 元，不确定在售的不推荐"
坏 prompt = "推荐几本书看看"

原则 6：复杂任务要拆成子任务（Decomposition）
-----------------------------------------------
模型有时候会自己拆步骤（尤其是 System Prompt 里写了或者框架帮拆了），
但不保证每次都拆。模型的智能是概率性的，工程系统必须是确定性的。
把一个大任务拆成多个小步骤，每个步骤用上一步的输出作为输入。
这会直接用在 Week 2 Day 4 的 Tool Calling 循环中。

原则 7：迭代比一次性完美重要
------------------------------
第一版 prompt 花 5 分钟写出 → 测试 → 看哪里坏了 → 修 → 再测试。
每次只改一个变量（只改角色/只改格式/只加一个例子），不要一次改多个。
Prompt 也是代码，只是用自然语言写的——写完一段跑一段，不是写 500 行再点运行。

原则 8：输出格式必须明确到"没有第二种写法" ★
----------------------------------------------
格式模糊 = 输出无法解析 = 程序崩溃。
"输出简介" → 模型写 50 字也行、300 字也行。
"输出简介，不超过 80 字，只含产品名称和核心功能，JSON 格式 {name: string, feature: string}"
→ 模型只能按这个来。
这直接关联 Week 2 Day 3 的结构化输出和 JSON Mode。

原则 9：不同模型对同一段 prompt 的反应不同
-------------------------------------------
GPT-4o 严格遵循约束，Claude 倾向理解意图而非逐字执行，DeepSeek 对中文敏感但英文约束偶有松弛。
所以 prompt 模板要考虑 provider 变量，不同模型可能需要微调格式。
关联 Week 2 Day 2 的 Jinja2 模板系统。

原则 10：用数据说话——"感觉好了"不算数 ★
------------------------------------------
主观判断无法复现、无法对比、无法改进。
正确做法：同一批 50 道测试题，3 种 prompt 各跑一遍，对比准确率、格式合规率。
用数据验证哪版更好，以及好在哪些题型上。
这直接关联 Week 2 Day 5 的评测框架——写评测脚本跑 50 条用例，输出评测报告。


三、我觉得最重要的 3 条原则
============================

选了第 2、8、10 条，理由：

原则 2（五要素结构）
  因为它是"骨架"。其他 9 条原则都是在这个骨架上的优化和补充。
  后面写任何 System Prompt，先把这五个要素填好，就对了 80%。

原则 8（格式明确到没有第二种写法）
  因为它是"工程底线"。Agent 不是给人类看的，输出要喂给下一个程序。
  格式不一致 = 系统崩。这直接决定了 Agent 能不能真正跑起来。

原则 10（用数据说话）
  因为它是"手艺升级为工程的标志"。主观感觉无法迭代、无法对比、无法证明。
  能用准确率、成功率这种数字说话，才算真正的工程师。


四、今日核心收获
=================

1. LLM 的本质是概率系统，不是思考系统。Prompt 的作用是"收窄概率分布"。
2. 好的 System Prompt = 角色 + 上下文 + 任务 + 约束 + 输出格式，五要素缺一不可。
3. Prompt 是代码——要迭代、要测试、要用数据说话，不是写一次就完事。
4. 不同模型对 prompt 的反应不同，写模板时要考虑模型差异。
