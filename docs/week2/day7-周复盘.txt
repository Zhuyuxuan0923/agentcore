二、每日回顾
============

Day 1 — Prompt 工程理论
-----------------------
  学了什么：精读 Prompt Engineering Guide，提炼 10 条核心原则
  关键认知：
    - LLM 的本质是"逐 token 概率预测器"，不是思考系统
    - Prompt 的作用是"收窄概率分布"——越精确的 prompt，LLM 越不会乱猜
    - 五要素骨架（角色+上下文+任务+约束+输出格式）= 任何 system prompt 的模板
    - Chain-of-Thought 的原理：前面的推理 token 约束后面的结论 token
  TOP 原则：
    原则 2：五要素结构（这是"骨架"，其他原则都是优化）
    原则 8：格式必须明确到"没有第二种写法"（这是"工程底线"）
    原则 10：用数据说话——"感觉好了"不算数（这是"手艺升级为工程的标志"）


Day 2 — Prompt 模板系统（Jinja2）
-------------------------------
  学了什么：用 Jinja2 实现可变 prompt 模板，支持 Few-Shot 动态注入
  关键认知：
    - 模板引擎 = "邮件合并"——底稿不变，变量填空
    - {{ }} 是输出值，{% %} 是控制逻辑（if/for）
    - 示例和模板要分离：示例是"数据"，模板是"结构"
    - 标签筛选 + 限制数量 = 用最少 token 获得最大示范效果
  踩过的坑：
    - 模板变量设计反了——把"不变的部分"做成了变量，导致模板到处都是 {{ }}
      -> 正确做法：只把"会变的"做成变量，固定内容直接写死在模板里
    - Few-Shot 示例没按标签筛选，全部塞进去 → token 爆炸
      -> 加了 filter_by_tags() + pick(max_count=3)
    - code_review.j2 里 {% if few_shot_text %} 忘了写 → 不传示例时模板渲染空白


Day 3 — 结构化输出（JSON Mode + Tool Use）
-----------------------------------------
  学了什么：三种方案让 LLM 稳定输出 JSON
  关键认知：
    - LLM 不是"先想好 JSON 再填空"，而是边想边写——偶尔写出非法 JSON 是数学必然
    - JSON Mode = "写字用田字格"（保证是字，不保证内容）
    - Tool Use = "填表"（表头定了，LLM 只能按表头填）
    - Function Call ≈ Tool Use，只是用的 OpenAI 的函数调用机制
    - 成功率排序：Function Call > JSON Mode >> 纯 Prompt
  踩过的坑：
    - 以为 JSON Mode 就够 → 字段经常漏 → 补了 system prompt 里的 Schema 描述
    - Anthropic 没有 JSON Mode → 只能用 Tool Use → 理解了"不同模型不同方案"
    - _parse_json_response 只写了 3 种抢救策略 → 补到 5 种（加了无标记代码块 + 首尾花括号）


Day 4 — Tool Calling 完整循环
-----------------------------
  学了什么：实现 LLM 决策->执行工具->结果回传->再决策的完整循环
  关键认知：
    - Agent 的"大脑" = Tool Calling 循环
    - 消息历史累积是循环的核心——每次工具调用都追加到 messages
    - 两种协议本质相同（告诉 LLM "你可以调这些函数" -> 返回调用请求 -> 执行 -> 回传）
    - 安全是工具设计的绝对底线——绝不给 LLM eval() 或 shell 权限
  踩过的坑：
    - tool_loop 里 OpenAI 和 Anthropic 的消息格式不统一 → 按 sdk_type 分派两个方法
    - 工具执行出错时没回传错误信息给 LLM → LLM 不知道失败了，一直重试
      -> 把错误包装成字符串喂回去："工具执行失败：{error}，请换一种方式"
    - 循环忘了设上限 → 加了 max_rounds=5
    - Calculator 直接用了 Python eval() → 改成 ast.parse() 白名单模式（三层防御）


Day 5 — Prompt 评测框架
-----------------------
  学了什么：写评测脚本，用 50 条测试用例对比 3 种 prompt 写法的准确率
  关键认知：
    - 评测框架 = 把"感觉"变成"数字"——精确到"category 准确率 90% vs 70%"
    - 四个评测维度：JSON 合法性 -> 字段完整性 -> 逐字段准确率 -> 综合通过率
    - Prompt 风格差异在边界 case 上最明显——简单题大家都能对
    - 50 条用例 x 3 种风格 = 150 次调用 = 对照实验的统计学基础
  踩过的坑：
    - 第一版 minimal prompt 连 category 有哪些选项都没说 → LLM 自创分类名
      -> v2 至少列出 5 个选项名
    - structured 版本 category 准确率只有 70% → 加了"分类决策树"
      -> v2 提升到目标
    - fewshot 版本的范例选了"简单题" → 对边界 case 帮助不大
      -> 换成 4 个"容易搞混的边界 case"作范例
    - evaluator 每次调用前重建 StructuredExtractor → 浪费初始化时间
      -> extractor 提到 evaluate_one 外面，复用同一个实例


Day 6 — System Prompt 设计实战
-------------------------------
  学了什么：用评测框架对比 3 种 prompt 设计哲学
  关键认知：
    - 反例（Negative Examples）是性价比最高的优化手段
      -> 6 个反例共 300 字，category 准确率 100%
    - Chain-of-Thought 不是万能药——在 deepseek-chat 上反而降准确率
      -> 理论正确不代表在所有模型上都有效
    - "说清楚为什么"（explain_why）对 category 判断有帮助，对 sentiment 帮助不大
      -> 情感判断更多依赖对语气的敏感度，不是对业务的理解
    - 最佳 prompt 是组合拳，但要注意 token 预算（500-1500 字最佳）
  实测数据（Quick 模式，deepseek-chat）：
    1st negative_examples     通过率 80%  字段: category 100% priority 90% sentiment 90%
    2nd explain_why           通过率 60%  字段: category 90%  priority 100% sentiment 70%
    3rd reasoning_template    通过率 30%  字段: category 80%  priority 70%  sentiment 70%

四、踩坑 TOP 10（写文章素材）
=============================

以下是在本周学习过程中遇到的最有价值的坑，每个都可以写进文章。

1. Prompt 不列出分类选项 → LLM 自创 category 名称
   现象：只说"分类"，没给具体选项。LLM 输出 "payment"、"tech_support"
        而不是 "billing"、"technical"
   教训：永远给出完整的候选集合。不只说"分类"，要说"从以下五项选择：
        billing/technical/account/product/complaint"
   延伸：这和 Day 1 原则 8 直接关联——格式明确到"没有第二种写法"

2. Few-Shot 范例选了"简单题"而不是"边界 case"
   现象：3 个范例都是标准 billing / technical / account 的典型 case，
        评测结果对边界 case 帮助不大
   教训：范例的价值不在于"覆盖简单情况"，而在于"展示你在边界上的判断偏好"
        简单题 LLM 本来就会，给它看简单题 = 浪费 token
   正确做法：选 4 个"容易搞混的边界 case"——支付失败、抱怨扣费、问订阅、真正的投诉

3. 认为 JSON Mode 就够了，没在 system prompt 里重复 Schema
   现象：JSON Mode 确实保证输出是合法 JSON，但字段经常漏
   教训：JSON Mode 只保证"是 JSON"，不保证"包含你想要的字段"
        要在 system prompt 里再描述一遍 Schema（双重保险）
        这就是"纵深防御"——API 层锁一道，prompt 层锁一道

4. 工具执行用了 eval()，给 LLM 留了安全后门
   现象：calculator 用 Python 的 eval() 计算表达式
        如果 LLM 输出 __import__('os').system('del /f /q *')，eval 直接执行
   教训：永远不要 eval() LLM 的输出。用 ast.parse() 先解析为抽象语法树，
        遍历检查只允许安全节点，再用 compile() + exec() 在白名单环境中执行
   延伸：这是 Agent 安全的第一课——LLM 是不可信的，要假设它可能输出任何东西

5. 循环忘了设上限 → LLM 可能陷入无限循环
   现象：tool_loop 写 while True，LLM 在"查数据 -> 不满意 -> 再查"之间反复
   教训：所有 while 循环必须有退出条件。Tool Calling 循环用 max_rounds=5
   延伸：每轮都在消耗 token（钱），无限循环 = 无限烧钱

6. 工具执行失败时没告诉 LLM → 它不知道失败了
   现象：calculator 收到 "11*"（不完整表达式）报错，但程序只记录了错误，
        没把错误信息回传给 LLM。LLM 看到"工具调用无响应"就重试同一个表达式
   教训：工具执行结果（包括错误）必须回传给 LLM
        正确做法：把异常包装成字符串 "工具执行失败：{error}，请修正表达式后重试"

7. Chain-of-Thought 在 deepseek-chat 上反而降准确率
   现象：给 reasoning_template prompt（先分析再输出 JSON），
        deepseek-chat 的通过率只有 30%，远低于 negative_examples 的 80%
   教训：技术在论文里有效 ≠ 在你的模型上有效
        不同模型对"混合输出格式"（分析文字 + JSON）的处理能力不同
        每次引入新技术（CoT / Few-Shot / 结构化推理），都要跑评测验证

8. 评测时只看总体通过率，没看逐字段准确率
   现象：看通过率 70%，觉得"还不错"
        但拆开看发现：category 90%、priority 85%、sentiment 55%
        -> 其实是 sentiment 拖了后腿，其他两个还不错
   教训：总体指标会掩盖局部问题。评测报告必须拆到逐字段——哪个字段差、差多少
   正确做法：evaluator.summarize() 输出 field_accuracy，逐字段对比

9. 模板里把"固定内容"做成了变量 → 模板到处都是 {{ }}
   现象：code_review.j2 第一版有 12 个变量，其中 6 个从来不改
   教训：模板变量 = 只把"会变的"做成变量。固定的内容直接写死在模板里
        变量太多了反而更难维护——调用 render 时要传一堆"其实不改"的参数

10. 不同厂商的 Tool Calling 格式不统一 → 需要双协议支持
    现象：OpenAI 的 tool_result 是独立的 role:"tool" 消息，
         Anthropic 的 tool_result 包在 user 消息的 content 块里
        如果只写一套逻辑，换厂商就崩
    教训：抽象层的核心价值就是"消化差异"
         ToolCallLoop 按 sdk_type 分派到 _run_openai_loop / _run_anthropic_loop
        外部调用者只看到 run()，看不到两个协议的区别
