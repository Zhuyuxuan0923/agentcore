一、为什么选 Next.js + shadcn/ui？
======================================

▶ Next.js 是什么？

  Next.js 是 React 的一个"框架之上的框架"。React 本身只管 UI 组件，
  Next.js 额外提供了：
    - 路由（文件放进 app/ 目录自动变成 URL 路径）
    - 服务端渲染（首次加载更快、SEO 友好）
    - 开发体验（Turbopack 热更新，改代码秒级刷新）

  简单说：React 是造砖的，Next.js 是用砖盖房子的建筑标准。

▶ shadcn/ui 是什么？

  它是一个"复制粘贴式"组件库，和 Ant Design / Element UI 有一个关键区别：

  传统组件库（Ant Design）：
    npm install antd → node_modules 里多几百兆 → 全局引入样式
    想要一个按钮？import { Button } from 'antd'
    改按钮样式？通常很痛苦——你要覆盖全局 CSS。

  shadcn/ui：
    npx shadcn add button → 在 components/ui/ 下生成一个 button.tsx
    源码在你项目里，你想怎么改就怎么改，不依赖 node_modules。
    相当于给你一个精装修的房子，你可以随意砸墙改造。

▶ Tailwind CSS

  传统的 CSS 写法：写 class 名 → 在 .css 文件里定义样式
  Tailwind：class 名就是样式本身

  例：
    传统写法：.my-button { background: blue; color: white; padding: 8px 16px; }
             <button class="my-button">提交</button>

    Tailwind：<button class="bg-blue-500 text-white px-4 py-2">提交</button>

  好处：不用在 HTML 和 CSS 之间来回切，一个文件就能看到所有样式。
  坏处：class 名很长。但习惯了反而更快。


二、项目结构——Next.js App Router
==================================

  frontend/
  +-- src/
  |   +-- app/                    # 路由（文件即路径）
  |   |   +-- layout.tsx          # 全局布局（<html> + <body> 壳）
  |   |   +-- page.tsx            # 首页（/ 路径对应的内容）
  |   |   +-- globals.css         # 全局样式 + CSS 变量主题
  |   +-- components/
  |   |   +-- ui/                 # shadcn/ui 生成的组件
  |   |   |   +-- button.tsx
  |   |   |   +-- card.tsx
  |   |   |   +-- input.tsx
  |   |   |   +-- scroll-area.tsx
  |   |   |   +-- separator.tsx
  |   |   |   +-- textarea.tsx
  |   |   +-- chat/               # 业务组件
  |   |       +-- sidebar.tsx     # 左侧栏
  |   |       +-- chat-area.tsx   # 聊天区
  |   |       +-- input-area.tsx  # 底部输入
  |   +-- lib/
  |       +-- api.ts              # API 客户端
  +-- package.json
  +-- next.config.ts

▶ layout.tsx 和 page.tsx 的区别

  layout.tsx 是"画框"——定义整个页面的骨架（<html> + <body>），
  里面的 {children} 会被替换成 page.tsx 的内容。
  切换页面时，layout 不重新渲染，只有 children 变。

  page.tsx 是"画的内容"——/ （首页）对应的实际内容。
  如果你以后加 /settings 页面，就创建 src/app/settings/page.tsx。


三、Warm Scholar 暖色调主题
==============================

▶ 设计方向：不是那种蓝紫色 AI 风格

  市面上的 AI 产品几乎全是"白底 + 紫色渐变 + Inter 字体"——
  你一看就知道这是 AI 生成的 UI，没有性格。

  这次选的方向是"暖色调学者风"（Warm Scholar）：
    - 底色：奶油米白（oklch(0.975 0.008 85)），像旧书的纸
    - 强调色：琥珀棕（oklch(0.55 0.12 55)），像墨水的颜色
    - 字体：衬线体标题 + 等宽代码字体
    - 纸纹理：SVG 噪声滤镜覆盖，模拟纸张的微颗粒感

▶ oklch 颜色空间——为什么不用 #RRGGBB 或 rgb()？

  #RRGGBB（十六进制）是人类历史上第一个 CSS 颜色格式，但非常不直观：
    #faf7f2 是什么颜色？你脑子里没法算出它是"暖白"。

  oklch 是 2024+ 的新标准，用三个更符合人类感知的参数定义颜色：
    L（亮度 Lightness）: 0=黑, 1=白, 0.975=很亮的暖白
    C（饱和度 Chroma）: 0=灰, 值越大色彩越鲜艳, 0.12=淡淡的琥珀色
    H（色相 Hue）: 角度制, 55=橙黄, 180=青, 270=紫

  比较好理解：oklch(0.975 0.008 85) → 亮 + 极淡 + 偏橙 = 暖白纸色
            oklch(0.55 0.12 55)  → 中等亮 + 中饱和 + 橙黄 = 琥珀墨水色

▶ 纸纹理的实现

  body::after 伪元素 + SVG 噪声滤镜：
    - position: fixed 覆盖整个页面
    - opacity: 0.015（几乎看不到，但细看有颗粒感）
    - pointer-events: none（不阻挡点击）
    - SVG feTurbulence 生成柏林噪声

  这个技巧来自图形学的"噪声叠加"原理——在人感知不到的层面加纹理，
  让纯色看起来像真实材质（纸、布、石头）。iOS 和 macOS 的 UI 大量使用这个技巧。


四、三个业务组件拆解
======================

▶ 组件树

  page.tsx
  +-- Sidebar（左侧 w-64）
  |   +-- Logo（"PersonalQA"）
  |   +-- 知识库列表（radio 式选择）
  |   +-- 对话历史列表（可点击）
  |   +-- 新建对话按钮
  +-- ChatArea（中间 flex-1）
  |   +-- 空状态（引导文字 + 装饰引号）
  |   +-- MessageBubble（用户消息 + AI 消息）
  |   +-- 加载动画（弹跳点）
  +-- InputArea（底部）
      +-- 文件上传按钮
      +-- 文本输入框（Enter 发送）
      +-- 发送按钮

▶ Sidebar 的状态管理

  两个独立的数据列表：
    - 知识库列表：从 GET /api/kb 获取，可创建新知识库
    - 对话历史：从 GET /api/conversations 获取，可点击加载历史

  关键 prop：
    - refreshKey：当父组件触发刷新时（比如新对话创建了），
      Sidebar 检测到 refreshKey 变化 → 重新拉取列表

  为什么不用全局状态管理（Redux / Zustand）？
    这两个列表仅 Sidebar 使用，不需要全局共享。
    在局部组件用 useState + useEffect 就够了。
    等需要跨多个组件共享状态时再引入全局状态管理也不迟。

▶ ChatArea 的消息渲染

  用户消息：
    - 右对齐
    - 琥珀色淡背景的 Card 包裹
    - 标签"YOU"在左上角

  AI 消息：
    - 左对齐
    - 无背景卡，直接显示文本
    - 标签"PersonalQA"在左上角
    - 关键：内容中的 [N] 被替换为 CitationBadge 组件

▶ 引用 UI —— 两层设计

  第一层：文本内的引用标记（superscript 小标签）
    - [1] 被解析为带数字的圆形标签
    - 琥珀色背景（bg-accent/15）
    - cursor-pointer + hover 效果
    - hover 时弹出 tooltip 显示来源文本前 100 字

  第二层：消息下方的引用列表
    - 当消息有引用时，在文字下方显示"引用来源"区域
    - 每条引用：左侧琥珀色细线 + 编号圆形 + 来源文本
    - 作用：让用户一目了然所有出处

  为什么是两层？
    第一层（文本内）帮助用户快速定位"这句话来自哪里"。
    第二层（底部列表）给用户一个完整的信息源概览。
    两者配合，用户不需要来回翻找。

▶ 加载状态

  LLM 生成答案需要几秒到几十秒。这个等待时间必须给用户反馈：
    - 三个依次弹跳的小圆点（动画延迟不同，wave 效果）
    - 文字："检索中..."
    - 输入框 disabled（防止用户在等待时重复发送）

  如果完全没有加载状态，用户会以为"卡死了"。
  加载状态是 UI 设计中性价比最高的投入——几行 CSS 动画就能大幅改善体验。

▶ InputArea 的细节

  - 文件上传：隐藏的 <input type="file"> + 美化的 <label>
    这是标准做法——原生 input[type=file] 太丑，用 label 伪装的按钮触发
  - Enter 发送 / Shift+Enter 换行：一行代码 onKeyDown 处理
  - 上传反馈："已索引 xxx 文件（N 个文档块）" 或 "上传失败: 原因"


五、API 客户端
================

  frontend/src/lib/api.ts 是前端和后端的"合同"：

    前端调用      后端响应
    ────────────────────────────────────────────
    uploadFile()  →  POST /api/upload   →  UploadResponse
    chat()        →  POST /api/chat     →  ChatResponse
    listKBs()     →  GET  /api/kb       →  string[]
    createKB()    →  POST /api/kb       →  void
    deleteKB()    →  DELETE /api/kb/:name →  void
    listConversations() → GET /api/conversations → Conversation[]
    getHistory()  →  GET  /api/history/:id →  Message[]

  每个函数封装了 fetch() 调用 + 错误处理，组件不需要关心 HTTP 细节。


六、前后端数据流——一次"提问→回答"的全过程
=============================================

  用户输入 "什么是深度学习？"
         │
         ▼
  InputArea.handleSend()
         │ onSend(question)
         ▼
  page.tsx handleSend()
         │ 1. 添加 userMsg 到 messages
         │ 2. setIsLoading(true)
         │ 3. chat(question, kbName, conversationId)
         ▼
  api.ts chat()
         │ fetch POST /api/chat { question, kb_name, conversation_id }
         ▼
  FastAPI /api/chat
         │ → PersonalQAAgent.chat()
         │ → 检索 → 生成 → 引用解析
         │ → 返回 { answer, citations, conversation_id }
         ▼
  page.tsx
         │ 1. 添加 assistantMsg 到 messages
         │ 2. 更新 conversationId
         │ 3. 递增 sidebarRefreshKey→侧边栏重新拉取对话列表
         │ 4. setIsLoading(false)
         ▼
  ChatArea 重新渲染 → 显示带引用标记的答案


七、今天遇到的 Bug 及修复
============================

▶ Bug 1：新对话不显示在侧边栏的历史列表中

  根因：侧边栏 useEffect(fetch, []) 只在组件挂载时执行一次。
        用户发了消息后，后端创建了新对话，但侧边栏不知道。

  修复：
    - 后端 chat API 响应新增 conversation_id
    - page.tsx 检测到新 conversation_id 后递增 sidebarRefreshKey
    - 侧边栏依赖 refreshKey 变化重新拉取列表

▶ Bug 2：对话历史点不进去

  根因：侧边栏的对话列表项只有样式，没有 onClick 事件。

  修复：
    - sidebar.tsx 新增 onSelectConversation prop
    - 每个对话项添加 onClick → 调用 onSelectConversation(id)
    - page.tsx 实现 handleSelectConversation → 调 getHistory(id) → 加载消息到聊天区
    - 选中的对话高亮（和知识库选中一样的样式）


八、和 Day 2 的关系——前后端是同一枚硬币的两面
================================================

  Day 2 搭了"引擎"（Agent + RAG 管道 + API），Day 3 搭了"外壳"（聊天 UI）。
  没有引擎的外壳是空壳，没有外壳的引擎是裸机。

  把 Day 2 和 Day 3 放在一起看：

    前端                后端                  技术栈
    ─────────────────────────────────────────────────────
    文件上传按钮    →    POST /api/upload    FormData + FastAPI
    文本输入框      →    POST /api/chat      JSON + RAG Pipeline
    知识库列表      ←    GET /api/kb         ChromaDB
    对话历史        ←    GET /api/conversations  SQLite
    引用标记 [1]     ←    citations 数组        parse_citations()

  中间那条竖线就是 HTTP 协议——前端不关心后端用什么语言实现，
  后端不关心前端用什么框架渲染。只要 API 格式对得上，就能通。
  这叫"前后端分离"。


九、今天的核心认知
==================

1. shadcn/ui 和传统组件库的本质区别：源码归你 vs 黑盒。
   你可以随意改 button.tsx，不需要覆盖 node_modules。

2. oklch 比 #RRGGBB 好理解。L=亮度 C=饱和度 H=色相，三个数字就能猜出大概颜色。

3. 引用 UI 两层设计：文本内标记（快速定位）+ 底部列表（来源总览）。

4. 加载状态是 UI 设计中性价比最高的投入——几行 CSS 动画就大幅改善体验。

5. 前后端分离的核心是"API 合同"：前端只关心调用什么 URL、收什么 JSON，
   后端只关心接收什么参数、返回什么 JSON。

6. 局部状态管理优先。Sidebar 的数据只有 Sidebar 用，用 useState 就够了。
   不要上来就引入 Redux。
