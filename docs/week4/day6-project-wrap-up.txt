一、README 怎么写——一个好 README 的六个要素
=============================================

一个好的 README 应该让一个陌生人在 3 分钟内：
  1. 知道这个项目做什么
  2. 知道怎么跑起来
  3. 知道怎么用

▶ 六要素清单

  1. 一句话描述（标题下方第一句）
     -> "上传你的文档，用自然语言提问，获得带来源引用的答案"

  2. Demo 预览（ASCII 或截图）
     -> 让读者在看到代码之前先看到效果
     -> 截图最好，没截图就用 ASCII 对话模拟

  3. 快速开始（clone -> install -> configure -> run）
     -> 每一步一个命令，复制粘贴就能跑
     -> 最容易劝退人的地方就是"跑不起来"
     -> 所以要提供 .env.example，说明每个环境变量的含义

  4. 项目架构（ASCII 图或截图）
     -> 让人一眼看懂"数据怎么流"
     -> 不画架构图的 README 对技术面试官来说等于没写

  5. API 文档（表格式）
     -> 列出所有端点、方法、说明
     -> 给一两个请求/响应示例

  6. 设计决策（Why 部分）
     -> 为什么选 ChromaDB 而不是 Milvus？
     -> 为什么用 Hybrid Search？
     -> 为什么 SQLite 而不是 PostgreSQL？
     -> 这部分展示你的"技术判断力"，
        面试时被问到"为什么这么设计"你就能直接引用


二、.env.example —— 开源项目的必备文件
======================================

▶ .env 和 .env.example 的区别

  .env          = 你的真实配置（含 API Key），绝不能提交到 Git
  .env.example  = 模板文件（只有 KEY 名，值是占位符），提交到 Git

  类比：.env 是你的钱包（不能给别人看），
        .env.example 是空钱包的照片（告诉别人"你需要往里放什么"）。

  别人 clone 你的项目后：
    cp .env.example .env   # 复制模板
    # 编辑 .env，填入自己的 Key


三、Git 提交记录整理——commit message 怎么写
============================================

▶ 一个好 commit message 的结构

  第一行（50 字内）：做了什么
  空一行
  后面（可以多行）：为什么、怎么做的

  例：
    Week 4: PersonalQA Agent -- 作品 1 完整交付

    整合前三周的 LLM Client / Prompt 模板 / RAG 管道，
    搭建 FastAPI 接口 + Next.js 前端，61 个测试用例全部通过。

▶ commit 的几个好习惯

  1. 一个 commit 做一件事（不要"修了 A 顺便删了 B 还重构了 C"）
  2. 用现在时（"Add feature" 而不是 "Added feature"）
  3. 解释为什么，而不是解释做了什么（代码本身已经说明了"做了什么"）

▶ Pre-commit hooks 的工作

  你提交代码时，pre-commit 会自动运行：
    - Black -> 格式化代码（统一空格、换行、引号风格）
    - Ruff  -> 检查代码规范（行太长？变量命名不规范？）
    - MyPy  -> 类型检查

  如果 hook 失败了，commit 就不会创建，你需要先修好问题再提交。
  这是"门卫"机制——不好的代码不能进门。


四、MyPy 和 ChromaDB 类型兼容性问题
=====================================

今天修 pre-commit hook 时遇到的一个实际问题：

  MyPy strict 模式 + ChromaDB = 很多误报错误

  原因：ChromaDB 是 Python 库，但它的类型标注不规范。
        比如 collection.get() 的返回值标注里有 | None，
        但实际使用时会返回数据，导致 MyPy 检查出很多"可能为 None"的错误。

  这不是我们代码的问题，是库的类型定义不完善。

  解决方案：
    1. 对于核心业务代码：修复真正的类型问题
    2. 对于第三方库类型问题：用 # type: ignore 标记，或跳过检查
    3. 对于 Demo/学习脚本：排除出 MyPy 检查范围

  这个经历本身也是学习——你会遇到"工具报错但实际没错"的情况，
  需要学会区分"我的代码有问题"和"工具/库太严格"。

六、今天的关键认知
==================

1. 代码写完了不等于项目做完了。写 README 和整理 Git 不是"额外工作"，
   是让代码"能被人理解"的必要步骤。

2. README 是项目的名片。面试官可能只看 30 秒，没看懂就直接关掉。
   所以第一屏必须说清楚"这是什么"+"怎么跑"。

3. .env.example 是开源礼仪。不要让别人猜需要配置什么环境变量。

4. Git 提交记录是你的"工作日志"。以后回头看，好的 commit message
   能让你快速理解当时做了什么决策。烂的 commit message（"fix" "update"）
   以后自己都看不懂。

5. 不是所有工具报错都是你的问题。MyPy + 类型标注不完善的库会产生误报。
   学会区分"真错误"和"库的类型定义不完善"。

6. 部署不是越早越好。Week 4 的项目是学习型项目，重点在理解原理。
   部署留到 Week 8+ 的正式作品上，那时候做部署更有价值。
