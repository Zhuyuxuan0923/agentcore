一、什么是 Agent？
===================

回顾前三周的技术栈：

  Week 1: LLMClient     ->  怎么和 LLM "说话"
  Week 2: Prompt 模板    ->  怎么"好好说"
  Week 2: Tool Calling  ->  怎么让 LLM "动手干活"
  Week 3: RAG 全链路    ->  怎么让 LLM "查资料后再说"

Agent 就是把上面四个捆在一起：

  Agent = LLM + Tools + Memory + Planning

  大脑(LLM) + 双手(Tools) + 记忆(Memory) + 规划(Planning)

你的第一个 Agent 包含：LLM + RAG(知识库) + 对话历史(Memory的雏形)

五、功能边界：什么做，什么不做
===============================

已确定的功能清单：

  本周要做的（核心功能）：
    1. 文件上传 — 支持 PDF、DOCX、Markdown 三种格式
    2. 文档处理管道 — 解析 -> 分块 -> 向量化 -> 存入 ChromaDB
    3. 问答 API — 提问 -> 检索 -> Prompt -> LLM 生成 -> 带引用返回
    4. 对话历史 — 用 SQLite 存储每轮对话
    5. 多知识库 — 可以创建、切换不同的知识库
    6. 前端界面 — 聊天式 UI，Markdown 渲染 + 代码高亮
    7. 引用可追溯 — 每个回答标注 [1] [2]，对应到具体文档片段

  本周不做的（留给以后）：
    - 用户登录/注册 — 先做单用户
    - 实时协作
    - 图片/表格识别 — PDF 里的图片先跳过
    - 网页抓取 — 只做本地文件上传
    - 邮件发送 — 留给 Week 5+ 的 Tool Calling

七、系统架构
============

 数据流设计：

                          用户
                           |
              +------------+-----------+
              |                        |
              v                        v
        [上传文件]                [输入问题]
              |                        |
              v                        v
      +---------------+        +---------------+
      |  FastAPI      | <----> |  Next.js      |
      |  后端服务器    |        |  前端界面      |
      +---------------+        +---------------+
              |
  +-----------+-----------+----------------+
  |           |           |                |
  v           v           v                v
[文件解析器] [ChromaDB] [LLMClient]    [SQLite]
  |          向量数据库   调用LLM         对话历史
  v
[文档分块管道]
PDF/DOCX/MD -> 纯文本 -> 分块 -> Embedding向量化 -> 存入ChromaDB


  路径A -- 上传文档（写入）：
    文件 -> FastAPI 接收 -> 解析(DOCX/PDF/MD -> 纯文本)
         -> 分块(RecursiveCharacterTextSplitter)
         -> Embedding API 向量化
         -> ChromaDB 存储(collection=知识库名称)
         -> 返回 "上传成功，共索引 N 个文档块"

  路径B -- 提问回答（读取）：
    问题 -> FastAPI 接收 -> Embedding API 向量化
         -> ChromaDB 检索 Top-K 相似文档块
         -> 相似度过滤(低于阈值的丢弃)
         -> 拼接 RAG Prompt 模板
         -> LLMClient 调用 LLM 生成
         -> 引用解析(正则提取 [N])
         -> 返回 {答案, 引用列表} 到前端渲染


八、技术选型
============

技术栈及选择理由：

  后端框架：   FastAPI     — Week 1 学过；自带 /docs；异步支持
  向量数据库： ChromaDB    — Week 3 用过；嵌入式，不需额外部署
  LLM 调用：   LLMClient   — Week 1 封装好的，支持多家 provider
  Prompt：     Jinja2 模板  — Week 2 建立的模板系统
  前端框架：   Next.js     — SPA 聊天体验；Vercel 免费部署
  UI 组件：    shadcn/ui   — 基于 Tailwind CSS，默认风格就不丑
  对话存储：   SQLite      — 嵌入式零配置，单用户够用
  文件解析：   python-docx + PyPDF2 + markdown
  部署：       后端 Fly.io + 前端 Vercel（都有免费额度）
