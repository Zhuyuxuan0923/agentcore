RAG 系统踩坑实录：从 Chunking 到引用追溯
=============================================

作者：AI Agent 学习者 (Week 3 产出)
日期：2026-06-21

---
导语
---

想象一个场景：你公司有 20 条内部制度——报销流程、年假天数、Wi-Fi 密码。
新员工问："公司 Wi-Fi 密码是什么？" 你打开 ChatGPT 问同样的问题。

GPT 说："公司 Wi-Fi 密码通常是 admin123，或者贴在路由器背面。"

听起来合理，但它是编的。GPT 不知道你公司的真实规定。这就叫"幻觉"。

RAG（检索增强生成）就是给 LLM 装一个"公司知识库"——每次回答问题前，
先去知识库里翻出相关资料，再基于资料回答。从"凭记忆"变成"查资料"。

听起来很简单对不对？文档存进去，搜出来，喂给 LLM。

实际上，从第 1 行代码到能交付的 RAG 系统，你会踩进至少 10 个坑。
这篇文章记录了我用一周时间从零搭建 RAG 系统时遇到的所有坑，
每个坑都包括"看到了什么现象"和"正确的做法是什么"。

读完你会知道：RAG 不是"检索 + LLM"两个词那么简单。


---
第一部分：分块的坑（Chunking）
---

坑 1：chunk_overlap 设太小，关键信息被"切"成两半

现象：
  RAG 的第一步是把长文档切分成小块（chunk）。每个块是检索的"最小单位"。
  你设了 chunk_size=512，chunk_overlap=0。

  公司文档中有一段：
    "年假政策：入职满 1 年享有 5 天带薪年假，满 3 年 10 天，满 5 年 15 天。
     年假需提前一周向直属上级申请。"

  512 个字符切一刀，恰好在 "满 5 年 15 天。" 后面切开。
  前半段在 chunk-A，后半段在 chunk-B。

  用户问："入职满 3 年有多少天年假？"
  检索时 chunk-A 和 chunk-B 的相似度都不够高（各自只有半条信息），
  都没进入 Top-K。正确答案漏掉了。

原因：
  分块边界是死的，文档内容的"话题边界"是活的。信息恰好落在边界上时，
  会被切成两半，每一半都不完整。

正确做法：
  设 chunk_overlap 为 chunk_size 的 10-20%。
  如果 chunk_size=512，overlap >= 50 个字符。

  overlap 让相邻块之间有"重叠区"——即使关键信息刚好落在边界上，
  至少有一个块会包含完整信息。

  代码对比：
    [错误] chunk_size=512, chunk_overlap=0
    [正确] chunk_size=512, chunk_overlap=80

  类比：用剪刀把报纸裁成小块贴进笔记本。如果每刀都切得严丝合缝，
  某一行字的上半部分在这一页，下半部分在下一页——两页都读不通。
  每刀重叠一行字，就能保证关键信息至少完整地出现在某一页上。


坑 2：只用一种分块策略，不管文档类型

现象：
  你用固定大小分块（Fixed-size Chunking）处理所有文档。
  对述职报告这种段落分明的文档还行，但遇到代码文件时——
  一个完整的函数被切成了 3 块，每块都是语法错误。

原因：
  不同文档有不同结构。代码有函数边界，对话有说话人切换，
  法律文件有条款编号。一刀切的策略不适合所有文档。

正确做法：
  根据文档类型选择分块策略：
    - 段落文本 -> 递归字符分块（按段落 -> 句子 -> 空格逐级切）
    - 代码文件 -> 语义分块（用 AST 找函数/类边界）
    - 对话记录 -> 按说话人切分
    - 通用场景 -> 固定大小分块（最朴素、最快）

  Demo 中对比了三种策略的结果差异——同一个文档，三种策略的输出完全不同。


坑 3：chunk_size 太大，LLM 注意力被无关文本淹没

现象：
  你把 chunk_size 设成 2048，每个块都很大。
  用户问"Wi-Fi 密码是什么？"，检索召回了正确的文档块。
  但 LLM 回答的是"文档中没有 Wi-Fi 密码的相关信息"。

  检索是对的，为什么 LLM 还是答错了？

  因为那个 2048 字符的块里，前 1000 字讲的是访客网络和 VPN 配置，
  Wi-Fi 密码夹在中间第 1200 个字符的位置。LLM 有"Lost in the Middle"
  效应——长文本中间的信息容易被忽略。

原因：
  LLM 的注意力不是均匀分布的。开头和结尾的信息被关注最多，
  中间的信息被关注最少。块太大，关键信息被周围的大量无关文本"埋"了。

正确做法：
  - chunk_size 设 256-512（经验范围），让每个块的信息密度更高
  - 最重要的检索结果放在 Prompt 的开头或结尾，不要排在中间
  - 如果必须用大 chunk，在 Prompt 里反复强调关键信息的位置


---
第二部分：检索的坑（Retrieval）
---

坑 4：只用向量检索，忘了专有名词需要关键词

现象：
  你用了向量检索（Cosine Similarity），信心满满。
  用户问："Study@2026 是什么？"（这是公司 Wi-Fi 密码）

  检索结果：报销流程、年假政策、入职手续...全是风马牛不相及的文档。
  相似度最高的也只有 0.32。

  为什么？因为"Study@2026"是一个专有字符串。在向量空间中，
  它没有语义相近的邻居——它就是一个"特殊字符串"。

原因：
  向量检索依赖语义相似度。"password"和"密码"在向量空间中很接近，
  但"Study@2026"作为一个随机字符串，没有语义，向量检索失效。
  而关键词检索（BM25）可以直接精确匹配 "Study@2026"。

正确做法：
  混合检索（Hybrid Search）= 向量检索 + 关键词检索。
  用 RRF（倒数排名融合）把两个排名合并。

  RRF 算法（10 行核心逻辑）：
    score[doc] = 0
    for each ranking in [vector_ranking, keyword_ranking]:
        for rank, doc in enumerate(ranking):
            score[doc] += 1.0 / (k + rank)  # k 通常取 60

  概念：在向量排名里排第 1 -> 贡献 1/61 = 0.016
        在关键词排名里排第 5 -> 贡献 1/65 = 0.015
        总分 0.031，可能比"向量排第 2 但关键词不在前 10"的文档高

  向量检索负责"意思相近的文档"，关键词检索负责"精确匹配的文档"。
  两者互补，不是取代。


坑 5：相似度阈值设太高，正确答案被过滤掉

现象：
  你设了 sim_threshold=0.85，觉得"只给 LLM 看高度相关的文档"。
  用户问"出差去北京能报销多少住宿费？"，系统回复"无法回答您的问题"。

  实际上知识库里就有这条信息（"一线城市住宿标准 500 元/晚"）。
  但检索的相似度是 0.78，被 0.85 的阈值过滤掉了。

原因：
  Embedding 模型不是完美的。500 字和 20 字的文本，即使内容完全一样，
  向量相似度也不会是 1.0。阈值过高 -> 文档被误杀。

正确做法：
  - 0.6 是一个安全的起点（Day 6 的评测数据支持这一点）
  - 如果场景对精度要求极高（医疗、法律），可以调到 0.7-0.75
  - 如果场景对覆盖率要求高（客服、百科），可以降到 0.4-0.5
  - 没有"正确"的阈值，只有"适合你场景"的阈值——跑评测数据来定


坑 6：检索结果太多，不想关的文档"污染"了 Prompt

现象：
  你设了 Top-K=10，觉得"给 LLM 看的多，答案越靠谱"。
  用户问"年假有多少天？"检索返回了 10 条文档。

  LLM 回答：
  "根据资料 [7] 代码提交规范，年假天数取决于你的入职年限...
   根据资料 [1] 出差报销标准，年假天数..."

  LLM 被无关文档混淆了。代码提交规范和报销标准跟年假关系不大，
  但 LLM 看到它们出现在参考资料列表里，就试图从中提取"关于年假"的信息。

原因：
  Top-K 太大 -> 无关文档混入 -> LLM 被误导。
  LLM 默认假设"你给的资料都是相关的"，不会主动判断哪些资料无关。

正确做法：
  Top-K 一般设 3-5。对于简单问题（答案只在一个文档中），Top-3 足够。
  对于复杂问题（需要综合多条资料），Top-5 到 Top-7。
  绝不能因为"想更全面"就把 Top-K 设成 10+——会引入大量噪音。


---
第三部分：生成的坑（Generation）
---

坑 7：没在 Prompt 里要求引用，答案对但没法验证

现象：
  用户问"入职满 3 年有多少天年假？"
  LLM 回答："入职满 3 年的员工享有 10 天带薪年假，年假需提前一周申请。"

  答案是对的。但用户不知道——这是真的吗？还是 LLM 编的？

原因：
  RAG 的完整链路是：检索 -> 增强 Prompt -> 生成 -> 引用。
  如果你只做了前两步（检索 + 增强），没要求引用，用户就无法验证信息来源。

正确做法：
  在 RAG 的 Prompt 模板中加入引用指令：
  ```
  ## 回答要求
  1. 只能基于以上参考资料回答问题
  2. 每个关键事实陈述后面必须标注来源编号，例如 [1]、[2][3]
  3. 如果参考资料中没有相关信息，直接说"根据现有资料无法回答"
  ```

  这样 LLM 的输出变成：
  "入职满 3 年的员工享有 10 天带薪年假 [1]。年假需提前一周申请 [1]。"
  用户看到 [1]，就能追溯到"年假政策"这条原始文档。

  引用标注 = 连接"答案"和"来源"的桥梁。没有它，RAG 只做了一半。


坑 8：忘了写兜底规则，LLM 对"不知道"的问题编造答案

现象：
  用户问"公司附近有什么好吃的餐厅？"
  知识库里全是公司制度文档，没有餐厅信息。
  检索结果相似度都很低（最高 0.25）。

  但你忘了在 Prompt 里加兜底规则。LLM 看到几条不相干的制度文档后：
  "根据公司招待费报销规定，人均标准不超过 200 元。
   建议您可以在预算范围内选择附近的餐厅。"

  这比沉默更危险——LLM 把"招待费规定"歪曲成了"餐厅推荐"。

原因：
  LLM 被训练成"总是给出有帮助的回答"，它不会主动说"我不知道"。
  当检索不到相关文档时，你必须明确告诉它：可以说不知道。

正确做法：
  RAG Prompt 模板必须有兜底规则：
  "如果参考资料中没有相关信息，直接说'根据现有资料无法回答'，不要编造。"

  同时代码层面做双重防御：
    max_sim = max(hit['similarity'] for hit in results)
    if max_sim < 0.4:
        return "根据现有资料无法回答您的问题。"

第四部分：评测的坑（Evaluation）
---

坑 9：只凭"感觉"判断 RAG 好不好，没有量化指标

现象：
  你改了 chunk_size 从 512 到 1024，重新部署。
  感觉"好像答案变差了"——但不知道差在哪，也不知道差了多少。

原因：
  没有评测体系时，你无法客观比较不同配置的效果。
  新配置上线后，用户反馈变差时，你只能凭感觉调回来。

正确做法：
  建立分层评测体系，至少 4 个指标：

  1. 检索召回率（Recall@K）：正确答案的文档有没有被搜到？
     这是 RAG 的"地基"指标。召回率=0，后面全白费。

  2. 答案准确性（Keyword Hit）：LLM 的回答有没有包含关键事实？
     Ground Truth 中定义的"必须出现的关键词"命中了几个。

  3. 引用正确性（Citation Precision）：LLM 标的 [1][2] 编号，
     有多少确实指向了存在的文档？编造的引用编号是"幻觉引用"。

  4. 拒答率（Rejection Rate）：知识库覆盖不到的问题，
     LLM 有没有正确地说"不知道"？

  每次改动（chunk_size、Top-K、相似度阈值、Prompt 模板）后跑一遍评测，
  对比改之前的数据。问题出在哪个环节一目了然。


坑 10：用关键词匹配做评测时，同义词替代被判为错误

现象：
  你设了 expected_keywords=["500", "一线城市", "住宿"] 作为"正确答案标志"。
  LLM 回答："北京的住宿标准为 500 元/晚"。

  关键词"一线城市"没命中——LLM 说的是"北京"。
  Keyword Hit Rate = 66%(2/3)，但实际上答案完全正确。

原因：
  关键词匹配是机械的字符串比较，不懂"北京是一线城市"这个常识。
  LLM 用了同义表达但被判错。

正确做法（分阶段）：
  学习阶段（现在）：关键词匹配足够帮你理解评测逻辑。
  生产阶段（Week 10）：升级到"LLM-as-Judge"——让 GPT-4 给另一个 LLM
  的回答评分，包括事实准确性、完整性、相关性等维度。
  这是 DeepEval / RAGAS 等评测框架的核心方法论。


---
总结：RAG 的 5 关通关清单
---

搭建 RAG 系统的过程可以拆成 5 关。每一关都有坑，但知道坑在哪就不会掉进去：

  [第 1 关] 分块（Chunking）
    -> 重叠够不够？策略对不对？大小合不合适？

  [第 2 关] 检索（Retrieval）
    -> 向量+关键词混合了吗？阈值合不合理？Top-K 会不会太多？

  [第 3 关] 生成（Generation）
    -> Prompt 里要引用标注了吗？写了兜底规则吗？

  [第 4 关] 引用（Citation）
    -> 引用格式统一了吗？解析逻辑兜底了吗？

  [第 5 关] 评测（Evaluation）
    -> 4 个指标都测了吗？改了配置后重新跑了吗？

这 5 关都通过，你的 RAG 系统就从一个"Demo"升级成了"能交付的工程"。

RAG 不是"检索 + LLM"两块拼图——它是从数据到答案的完整流水线。
踩完这 10 个坑之后，你再看任何 RAG 系统，都会自然地想：
"chunk_size 多少？overlap 多少？用的是混合检索吗？引用标注了吗？评测数据呢？"

这就是从"会搭 RAG"到"理解 RAG"的转变。
