一、"一个关键词打败了向量"——先看一个让你意外的例子
=====================================================

回顾 Day 2 你做的语义搜索：
  问 "生病了怎么请假" → 向量找到 "病假规定：请病假需提供..."
  问 "出差住酒店报销" → 向量找到 "出差报销标准：一线城市..."

向量检索很聪明——它知道"生病"和"病假"语义相关，"住酒店"和"住宿"同义。

但是，考虑这个问题：

  问："Wi-Fi 密码是什么？"

  知识库里有：
    文档 A: "办公网络配置：公司 Wi-Fi 密码为 Study@2026，支持 5G 频段。"
    文档 B: "网络安全培训：所有员工必须参加每年一次的信息安全培训..."

  向量检索："Wi-Fi"和"网络"语义接近 → A 和 B 的向量都靠前
  关键词检索："Wi-Fi"这个词在 A 出现，B 没出现 → 直接命中 A

  结果：关键词检索 win。因为专有名词"Wi-Fi"就是天然的最佳索引。

另一个例子：

  问："我发烧了，怎么请假？"

  知识库里有：
    文档 A: "病假规定：请病假需提供二级以上医院开具的病假证明。"
    文档 B: "年假政策：入职满 1 年享有 5 天带薪年假..."

  向量检索："发烧"和"病假"语义相关 → 找到 A
  关键词检索：找不到"发烧"这个词 → 可能匹配到无关文档（"发"出现在"薪资发放"）

  结果：向量检索 win。因为用户用了口语化表达，和正式文档用词不同。

▶ 核心洞察（今天最重要的认知）：

  向量检索 = 语义理解 → 擅长"意会"（同义词、改写、跨语言）
  关键词检索 = 字面匹配 → 擅长"精确"（专有名词、编号、代码）

  它们不是竞争关系，是互补关系。混合检索 = 两个都要。


二、策略 A：纯向量检索（你已经会的，今天只是换个角度）
=======================================================

向量检索的原理你 Day 1-2 已经学过：
  问题 → Embedding → 问题向量 → 在 ChromaDB 中找余弦距离最近的文档向量

▶ 优点：
  1. 理解语义——"发烧"能找到"病假"，不需要词对词匹配
  2. 跨语言——英文问题能找到中文文档（如果 Embedding 模型支持多语言）
  3. 抗改写——"怎么提交代码"和"PR 流程"指向同一件事

▶ 缺点：
  1. 专有名词是盲区——"Wi-Fi"和"无线网络"语义一样，但"Wi-Fi"这个精确字符串的匹配
     在纯向量检索中没有特殊权重
  2. 数字/编号——"第 3.2 条"和"第 3.3 条"的向量可能极其接近，但含义完全不同
  3. Embedding 模型决定上限——模型没见过"你公司的内部术语"，这些词的向量就没意义

▶ 适用场景：
  - 用户用自然语言提问（"感冒了怎么请假？"而非"病假规定"）
  - 文档是叙述性的（文章、报告、手册）
  - 跨语言检索

在本 Demo 中，VectorRetriever 类封装了 ChromaDB 的语义搜索。
你今天不需要重写这部分——你已经在 Day 2 用过了。


三、策略 B：关键词检索——"古老但好用"的搜索引擎技术
=====================================================

▶ 它是什么？

在向量检索出现之前，搜索引擎（Google、百度、Elasticsearch）靠关键词匹配工作。
核心算法叫 TF-IDF（词频-逆文档频率）。

▶ 生活类比：

  你有一堆书，想找"哪本书关于 Python 的内容最多"。
  TF-IDF 做的事是：

  1. TF（词频）：这本书里 "Python" 这个词出现了几次？
     出现次数多 → 这本书可能和 Python 有关

  2. IDF（逆文档频率）：整个书架上有多少本书包含 "Python" 这个词？
     很多书都包含 → "Python" 是个常见词，区分度不高
     只有少数书包含 → "Python" 是个特殊词，区分度高

  3. TF × IDF → 最终得分

  举例：
    - "的" 在每本书都出现很多次 → TF 高但 IDF 低 → 最终得分低（没啥用的词）
    - "Wi-Fi" 只在网络相关的书中出现 → TF 中但 IDF 高 → 最终得分高（好关键词）
    - "报销流程" 在财务类书中出现多 → TF 高且 IDF 中 → 得分高

▶ 算法步骤（只看一遍，不需要背）：

```python
# Step 1: 分词（简单2-gram）
"报销流程" → ["报", "报销", "销", "销流", "流", "流程", "程"]

# Step 2: 算 TF（词在该文档中的频率）
TF("报销", 某文档) = "报销"在该文档出现了几次 / 文档总词数

# Step 3: 算 IDF（词在整个语料库中的稀有度）
IDF("报销") = log(总文档数 / 包含"报销"的文档数) + 1

# Step 4: TF-IDF
TF-IDF("报销", 某文档) = TF × IDF

# Step 5: 对查询也做同样的事，然后算查询 TF-IDF 向量和文档 TF-IDF 向量的余弦相似度
```

▶ 为什么不直接用 count（数词频）？

  如果只看词频，"的"、"是"、"公司"这类高频词会排在最前面。
  但它们对"区分文档"没有帮助。IDF 就是用来打压这些"废话词"的。

▶ 优缺点

  优点：
    1. 专有名词精准匹配——"Wi-Fi"、"Study@2026"、"第 3.2 条"
    2. 不需要 Embedding 模型——离线可用，零 API 成本
    3. 可解释——你能说清为什么这篇文档排第一（"因为它包含'Wi-Fi'这个词"）

  缺点：
    1. 语义盲——"发烧"找不到"病假"（字面上没有重叠）
    2. 分词依赖——中文分词不准 → 检索不准（"南京市长江大桥"是"南京市"还是"南京市长"？）
    3. 同义词无法识——"住宿" vs "住酒店"


四、策略 C：混合检索——RRF 融合
==================================

▶ 它解决什么问题？

  向量检索和关键词检索各有所长。混合检索的目标很简单：
  "把两个结果合并，让好的更好，让差的被淘汰。"

▶ RRF（Reciprocal Rank Fusion）——最简单的融合算法

  不用管两个检索器各自的分数尺度（向量分数 0.85 vs TF-IDF 分数 0.35 不可比），
  只看排名。

  公式：RRF_分数 = w_vec/(k + rank_vec) + w_kw/(k + rank_kw)

  参数：
    k = 60（平滑参数，防止 1/1 = 1.0 权重太高）
    w_vec = 向量检索权重
    w_kw = 关键词检索权重

  举例：

    文档 A: 向量排名 #1, 关键词排名 #10
      RRF = 0.5/(60+1) + 0.5/(60+10) = 0.0082 + 0.0071 = 0.0153

    文档 B: 向量排名 #3, 关键词排名 #2
      RRF = 0.5/(60+3) + 0.5/(60+2) = 0.0079 + 0.0081 = 0.0160

    → 文档 B 胜出，因为它在两个检索器都表现稳定。

  对比：如果只用向量检索，文档 A 会胜出（Rank #1 vs #3）。
        只用关键词检索，文档 B 会胜出（Rank #2 vs #10）。
        混合检索：文档 B 胜出——它在两个维度上都还行，总分最高。

▶ 为什么不用简单加权分数？

  "向量 0.85 × 0.5 + TF-IDF 0.35 × 0.5 = 0.60" 看起来很直接。

  但问题是：两种分数的"尺度"不同。
    - 向量相似度 0.85 和你 embed 模型训练数据有关
    - TF-IDF 的 0.35 和你的语料库大小有关
    - 0.85 不一定比 0.35 "好"——它们完全不可比

  RRF 用排名替代原始分数，绕过了"不可比"的问题。
  排名是通用的——#1 就是比 #2 好，不管分数是 0.99 还是 0.01。


Demo 分为 4 个部分，逐个观察：

▶ 第一部分：8 条查询 × 3 种策略的对比评测

  看每一行：
    [V] = 向量检索 → 看 sim 分数变化
    [K] = 关键词检索 → 看 doc 排名和向量有什么不同
    [H] = 混合检索 → 看是否融合了两者

  观察点 1：有些查询三者结果一样（如 Q1 报销类），有些查询三者结果不同。
           结果不同的时候，混合检索通常表现更稳定。

  观察点 2：汇总表中的 Recall@3/5 vs Precision@3/5。
           Recall 高 = 不漏（所有相关文档都找到了）
           Precision 高 = 不杂（返回的结果中真正相关的比例高）

  ▶ 什么是 Recall 和 Precision？

    假设有 3 篇文档和"报销"相关，检索器返回了 5 篇结果：

      检索结果：[A(相关), B(不相关), C(相关), D(不相关), E(不相关)]
      相关文档：[A, C, F（被漏掉了）]

      Recall@5  = 找到了几个相关的 / 总共几个相关的 = 2/3 = 66.7%
      Precision@5 = 返回结果中几个相关的 / 返回了几个 = 2/5 = 40%

      好的检索器：Recall 高（不漏）+ Precision 高（不杂）。
      但两者往往是跷跷板——返回更多结果 → Recall 上升，但 Precision 下降。

▶ 第二部分：向量 vs 关键词的"互补性"

  案例 1（语义改写）：
    查询 "我需要去医院看病，公司有什么规定？"
    "看病"这个词不在任何文档中，但向量知道它和"病假"相关。
    关键词检索找不到"看病"→ 可能匹配到无关文档。

  案例 2（精确匹配）：
    查询 "Wi-Fi 密码是 Study@2026 吗？"
    "Study@2026"是精确字符串，关键词检索直接命中。
    向量检索找到了一批"网络相关"文档，没有特别的区分度。

  ▶ 关键认知：真实的用户查询，有些偏语义，有些偏关键词。
    混合检索 = 两种都查，让 RRF 自动拼出最好的结果。

▶ 第三部分：RRF 权重调参

  五组权重（纯向量 → 纯关键词），看结果怎么变化。
  默认均衡(0.5/0.5)在大多数场景够用。
  如果你的知识库有大量专有名词（API 文档、法律条文），可以把关键词权重调到 0.7。

▶ 第四部分：检索策略选型决策树

  一个简单的决策树，帮你根据场景选策略。

六、BM25 —— 关键词检索的生产级升级
=====================================

本 Demo 的关键词检索用了简化版 TF-IDF，目的是让你理解原理。
生产环境中的关键词检索用的是 BM25（Best Match 25）。

▶ BM25 vs TF-IDF 的区别：

  | 维度 | TF-IDF（本 Demo） | BM25（生产环境） |
  |------|-------------------|-------------------|
  | TF 饱和 | 词出现越多分越高（线性） | 词出现 5 次和 50 次分差不多（饱和） |
  | 文档长度 | 长文占优（词多） | 惩罚过长文档（文档越长越难得高分） |
  | 参数调优 | 无参数 | k1（TF 饱和参数）+ b（长度惩罚参数） |
  | Python 库 | 本 Demo 手写 | rank-bm25 库（pip install rank-bm25） |

  为什么 BM25 更好？
    如果你搜"Python"，一篇 500 页的书在第 1 页提到一次，另一篇 2 页的博客全篇在讲 Python。
    TF-IDF 可能给 500 页的书更高分（因为"Python"出现了 50 次，但都是垃圾匹配）。
    BM25 会惩罚长文档，让 2 页的博客排得更前。

▶ 你现在不需要学 BM25，知道它的名字和作用就够了。做 Week 4 项目时如果需要，一行 pip install 即可。


七、三种策略的完整对比表
==========================

| 维度 | 向量检索 | 关键词检索 | 混合检索 |
|------|---------|-----------|---------|
| 擅长 | 语义相似、同义词、改写 | 精确匹配、专有名词、数字 | 两者兼顾 |
| 短板 | 专有名词、数字 | 同义词、语义 | 实现复杂一点 |
| 需要什么 | Embedding 模型（API 或本地） | 分词器（Jieba 或 2-gram） | 两者都需要 |
| API 成本 | 每次 Embedding 调用 | 零 | 每次 Embedding 调用 |
| 可解释性 | 低（"为什么选这篇？语义相近"） | 高（"因为包含'Wi-Fi'这个词"） | 中 |
| 离线可用 | 需要本地 Embedding 模型 | 是 | 需要本地 Embedding 模型 |
| 适用文档量 | 百万级（ChromaDB） | 十万级（高精度时） | 百万级 |
| 你这个阶段的推荐 | 会用，作为主力 | 理解，作为补充 | 推荐，Week 4 项目用 |


八、今天的核心认知
===================

1. 向量检索和关键词检索不是替代关系，而是互补关系。
   向量懂语义，关键词懂精确——混合检索两个都查，RRF 自动拼。

2. TF-IDF 是搜索引擎的基石算法。
   TF（词频）衡量"这文档在讲这个词吗"，
   IDF（逆文档频率）衡量"这个词有区分度吗"。

3. RRF（倒数排名融合）是融合多路检索结果的最简单方式。
   用排名而非分数 → 绕过不同检索器分数尺度不可比的问题。

4. Recall（召回率）和 Precision（精确率）是评价检索器的两个核心指标。
   召回 = 不漏，精确 = 不杂。两者往往是跷跷板。

5. 你的默认配置（可以变的）：
   混合检索 + RRF 均衡权重(0.5/0.5) + Top-K=5
