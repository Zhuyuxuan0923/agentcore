  5. 知道什么是 Ground Truth，以及为什么评测需要它

一、为什么要评测？——"差不多"不是工程
=====================================

▶ 生活类比：体检

  你说"我身体感觉差不多"——这不叫体检。
  体检是：量血压、抽血化验、拍 CT，每项都有数字，每个数字都有正常范围。

  RAG 也一样。"感觉回答得还行"不是评测。
  评测是：用 30 道标准题目跑一遍，每道题记录 4 个数字，看总分。

▶ 不评测的后果

  场景：你改了分块大小（chunk_size 从 512 改成 1024），重新部署。
  用户反馈"怎么答案变差了？"

  没有评测体系时：你只能凭感觉猜是哪里出了问题。可能要试半天才找到原因。
  有评测体系时：你跑一遍 30 题，对比改之前的得分 → 检索召回率下降了 15%。
  你马上知道：分块变大导致检索精度下降，某些关键信息被埋在大段文本里了。

  评测把"凭感觉调试"变成"看数据定位"。这是工程师和爱好者的分水岭。


二、4 个评测指标——每个回答什么问题？
=====================================

评测是分层级的——RAG 管道有"检索 → 生成 → 引用"三个环节，每个环节都要测到。

┌─────────────────┬──────────────────────┬──────────────────────┐
│     指标          │  测的是哪个环节？      │  通俗解释              │
├─────────────────┼──────────────────────┼──────────────────────┤
│ 检索召回率        │ 检索器（Retriever）    │ "正确答案的资料有没有   │
│ (Recall@K)       │                      │  被搜到？"             │
├─────────────────┼──────────────────────┼──────────────────────┤
│ 答案准确性        │ LLM 生成（Generator）  │ "LLM 的回答对不对？    │
│ (Keyword Hit)    │                      │  关键信息有没有提到？"  │
├─────────────────┼──────────────────────┼──────────────────────┤
│ 引用正确性        │ 引用解析（Citation）   │ "LLM 标的 [1][2]     │
│ (Citation Prec.) │                      │  是不是真对应了正确文档？│
├─────────────────┼──────────────────────┼──────────────────────┤
│ 拒答正确率        │ 边界处理              │ "知识库没有答案时，     │
│ (Rejection Rate) │                      │  有没有诚实说不知道？"  │
└─────────────────┴──────────────────────┴──────────────────────┘

▶ 指标 1：检索召回率 (Recall@K)

  定义：正确答案所在文档中，被 Top-K 检索结果覆盖的比例。

  公式：Recall@K = |检索到的正确答案文档| / |所有正确答案文档|

  例：
    正确答案在文档 [1, 2, 3] 中
    检索 Top-5 返回了文档 [1, 2, 8, 9, 10]
    → 召回了文档 1 和 2，漏掉了文档 3
    → Recall = 2/3 = 66.7%

  这意味着什么？
    检索是 RAG 的地基。如果 Recall = 0，正确答案根本不在检索结果里，
    LLM 再强也不可能回答对——它压根没看到正确资料。

  所以 Recall 是第一优先级指标。Recall 上不去，后面的指标都没意义。

▶ 指标 2：答案准确性 (Keyword Hit Rate)

  定义：期望关键词（Ground Truth 中定义的正确答案应包含的关键词）在 LLM 回答中出现的比例。

  公式：Keyword Hit Rate = 命中的关键词数 / 期望关键词总数

  例：
    期望关键词：["500", "一线城市", "住宿"]
    LLM 回答："根据公司规定，一线城市住宿标准为 500 元/晚 [1]。"
    → 命中了 "500" 和 "一线城市"（"住宿标准" 命中了 "住宿"）
    → Keyword Hit Rate = 3/3 = 100%

  为什么用关键词而不是让 LLM 当裁判？
    关键词匹配是你现在能理解、能手动验证的最简单方案。它不完美——
    同义词替换会漏判（LLM 说"首都"而不是"北京"），但它快速、可解释。

    生产环境通常用"LLM-as-Judge"——让 GPT-4 给另一个 LLM 的回答打分。
    这会在 Week 10 深入。

▶ 指标 3：引用正确性 (Citation Precision)

  定义：LLM 在答案中标注的引用编号 [N] 中，有多少指向了真实的、有效的文档。

  公式：Citation Precision = 有效引用数 / 总引用数

  例：
    LLM 回答："年假 10 天 [1]，试用期 3 个月 [5]。"
    检索到的文档：1→年假政策，2→病假规定，3→事假管理
    [1] 有效（source_map 中有 1）
    [5] 无效（source_map 中没有 5，LLM 幻觉引用）
    → Citation Precision = 1/2 = 50%

  引用是 RAG 的"良心"。答案对不对是一回事，但引用标注至少让用户能追溯。
  如果引用是错的（甚至编的），用户就去验证了一个不存在的来源——这比不引用更糟。

▶ 指标 4：拒答正确率 (Rejection Rate)

  定义：对于知识库覆盖不到的问题，系统是否正确地说"无法回答"。

  公式：Rejection Rate = 正确拒答数 / 超出范围题目总数

  为什么重要？
    如果用户问"公司附近有什么好吃的"——知识库里只有报销/请假/IT 制度——
    系统却说"公司食堂在 3 楼"——这就是幻觉，而且比"不知道"更危险。

  RAG 不是万能的。一个好的 RAG 系统知道自己不知道什么。


三、Ground Truth——评测的"标准答案"
===================================

▶ 什么是 Ground Truth？

  Ground Truth = 你预先定义的"正确答案应该包含的信息"。
  评测就是把系统的输出和 Ground Truth 做对比。

  每条测试用例需要：
    1. question: 用户问题
    2. ground_truth_docs: 包含正确答案的文档索引（测检索用）
    3. expected_keywords: 正确答案应包含的关键词（测答案准确性用）
    4. is_out_of_scope: 这个问题知识库能回答吗？（测拒答用）

▶ Ground Truth 怎么来？

  本次 Demo 的 30 条 Ground Truth 是手工标注的——对着 CORPUS 的 20 条资料，
  一条条写问题和对应的正确答案文档/关键词。

  生产环境中，Ground Truth 来自：
    - 真实用户日志（筛选出高频问题，人工标注标准答案）
    - 合成数据（让 LLM 根据文档库生成问题和答案，人工审核）
    - 领域专家标注（最贵但也最可靠）


四、配置对比——Top-K 和相似度阈值的 Trade-off
==============================================

▶ 为什么需要对比不同配置？

  RAG 管道有两个关键参数影响效果：

  1. Top-K（检索返回多少条文档）
     - K=3：只给 LLM 看最相关的 3 条 → 信息少但噪音也少
     - K=7：给 LLM 看 7 条 → 信息更全但无关内容也可能混入

  2. 相似度阈值（多相似的文档才纳入）
     - θ=0.4：宽松，几乎全要 → 召回高但噪音多
     - θ=0.8：严格，只要高分 → 精度高但可能漏掉正确答案

▶ Trade-off 的本质

    高召回   ←──────────────→  高精度
    (宽松)                      (严格)
    Top-K=7, θ=0.4            Top-K=3, θ=0.8

    宽松的好处：不容易漏掉正确答案
    宽松的代价：LLM 收到更多无关文本，可能被误导

    严格的好处：LLM 只看到高质量资料，答案更可靠
    严格的代价：可能过滤掉正确答案，导致 LLM 无资料可用

  没有"最好"的配置，只有"最适合你的场景"的配置。
  如果你的场景对准确性要求极高（如医疗、法律），偏向严格配置。
  如果你的场景对覆盖率要求高（客服、百科），偏向宽松配置。

▶ Demo 的对比设计

  跑 5 种配置组合：
    (Top-K=3, θ=0.4)  → 最宽松
    (Top-K=3, θ=0.6)  → 中等召回 + 高精度
    (Top-K=5, θ=0.6)  → 默认配置
    (Top-K=7, θ=0.6)  → 高召回
    (Top-K=5, θ=0.8)  → 最严格

  对比表中每项最优的标 ★，一眼看出哪种配置综合最好。


五、评测框架的代码结构
======================

Demo 脚本 w3d6_evaluation.py 的分层结构：

  ┌─────────────────────────────────────┐
  │ 第 1 部分：30 条 TestCase（含 GT）      │  ← 数据层
  ├─────────────────────────────────────┤
  │ 第 2 部分：rag_pipeline()            │  ← 被测系统（Day 5 同款）
  ├─────────────────────────────────────┤
  │ 第 3 部分：4 个指标的计算函数          │  ← 评测逻辑
  │  - compute_recall_at_k()             │
  │  - compute_keyword_hit_rate()        │
  │  - evaluate_citations()              │
  │  - is_rejection()                    │
  ├─────────────────────────────────────┤
  │ 第 4 部分：run_evaluation()          │  ← 运行器：遍历 30 题
  ├─────────────────────────────────────┤
  │ 第 5 部分：run_config_comparison()   │  ← 配置对比：5 种组合
  ├─────────────────────────────────────┤
  │ 第 6 部分：print_report()            │  ← 报告生成
  │  + export_report_json()             │     (终端 + JSON 双输出)
  ├─────────────────────────────────────┤
  │ 第 7 部分：离线模拟评测               │  ← 没 API Key 也能跑
  └─────────────────────────────────────┘

这种分层的好处：每一层只做一件事，改了测试数据不会影响评测逻辑，
改了评测逻辑不会影响报告输出。


六、离线模式——没有 API Key 也能理解评测
========================================

Demo 有一个"离线模拟模式"：当检测不到 LLM API Key 时，
用 generate_mock_answer() 模拟 LLM 的生成行为。

这个模式的价值：
  - 检索召回率和引用正确性的评测完全有效（不依赖 LLM）
  - 答案关键词命中率仅供参考（因为不是真正的 LLM 生成）
  - 可以让你在没有配置 API 的情况下先跑一遍评测框架，理解流程

当你配置好 API Key 后再跑，会看到真实的 LLM 生成效果。


七、常见评测结果解读
======================

▶ 场景 1：召回率 = 0

  问题："核心工作时间是什么时段？"
  检索结果：全是报销类文档，没有考勤相关文档。

  原因：检索器按主题分类索引，但"核心工作时间"没有被识别为"考勤"类。
  解决：检查检索器的查询分类逻辑 or 使用真正的语义向量（如 OpenAI Embedding）。

▶ 场景 2：召回率高，但关键词命中率低

  问题："入职满 2 年有多少天年假？"
  检索到了年假政策文档（召回成功），但 LLM 回答中没有提到"5 天"。

  可能原因：
    - LLM 没有准确提取文档中的数字
    - Prompt 模板对数字提取的指引不够强
    - 检索到的文档虽然相关但 LLM 没有正确理解

▶ 场景 3：引用精度 < 100%

  LLM 标注了 [5]，但检索结果只有 [1]-[4]。

  这说明 LLM 产生了"幻觉引用"——它编了一个不存在的引用编号。
  解决方案：
    - Prompt 中明确说"只能引用 [1] 到 [N]"
    - 代码层面过滤无效引用编号
    - 如果幻觉引用太多，降低 temperature 或切换到更强模型

九、今天的核心认知
===================

1. 评测是工程的基石。"感觉差不多"不能帮你定位问题，数据能。

2. 评测分层执行：检索 → 生成 → 引用 → 边界，每个环节独立评分。
   这样出问题时你能马上知道是哪个环节掉链子。

3. Recall 是第一优先级。正确答案文档都没检索到，后面的生成和引用毫无意义。

4. Ground Truth 是评测的前提。没有标准答案，就无从对比。

5. 引用正确性往往被忽视，但它决定了 RAG 的可信度。
   用户能追溯的答案才是可信的答案。

6. Top-K 和相似度阈值是互相制约的：调大能提高召回但增加噪音，
   调小能提高精度但可能漏掉答案。不同场景选不同配置。

7. 离线模式让评测框架在没 API 时也能跑——检索和引用评测不依赖 LLM。

8. 关键词命中率只是"快速评测"，生产环境会用 LLM-as-Judge。Week 10 会讲。
