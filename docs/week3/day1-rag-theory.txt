一、LLM 的先天缺陷——为什么需要 RAG？
======================================

在理解 RAG 之前，先理解它解决什么问题。

LLM 有两个你已经在 Week 1-2 体会到的问题：

问题 1：知识截止日期
    你问 GPT-4 "2026 年世界杯冠军是谁"，它不知道。
    因为它的训练数据截止到某个时间点，之后发生的事情它完全不知道。

问题 2：幻觉（Hallucination）
    你问 "我们公司内部的报销流程是什么"，它不知道，但它会编。
    编得很像真的——这就是幻觉。因为 LLM 的本质是"预测下一个 token"，
    不是"查询知识库"。当它不知道答案时，它不会说"不知道"，
    而是生成概率最高的 token 序列——哪怕这个序列是错的。

问题 3：私有知识盲区
    你花两周写了一份内部技术文档，LLM 完全不知道这份文档的存在。
    想让 LLM 基于你的私有文档回答问题？直接把文档内容塞进 prompt？
    一篇 5000 字的文档没问题，但如果是 500 篇、每篇 10000 字呢？
    GPT-4 的上下文窗口是 128K tokens（约 9.6 万中文字），看起来很大，
    但塞 50 篇文档就满了，而且 token 成本随上下文线性增长。

RAG 就是解决这三个问题的方案。


二、什么是 RAG？——用一个生活类比
===================================

RAG = Retrieval-Augmented Generation（检索增强生成）

想象你在参加一场**开卷考试**：

┌─────────────────────────────────────────────────────────┐
│                                                         │
│   闭卷考试（纯 LLM）         开卷考试（RAG）              │
│   ────────────────          ──────────────               │
│   你空手进考场               你带着一摞参考书进考场        │
│   遇到不会的题→只能瞎猜      遇到不会的题→翻书找答案       │
│   猜对了运气好               找到相关内容→基于内容作答     │
│   猜错了零分                 有据可查，还能标出处          │
│                                                         │
└─────────────────────────────────────────────────────────┘

"开卷考试"的流程分三步：

  第一步：翻书 → 找到和题目相关的段落     ← 这叫 Retrieval（检索）
  第二步：把找到的段落摆在眼前            ← 这叫 Augmented（增强上下文）
  第三步：看着段落写答案                  ← 这叫 Generation（生成）

RAG 就是把 LLM 从"闭卷考试"变成"开卷考试"的技术。

四、概念 1：Embedding（向量嵌入）——"把文字变成数字坐标"
==========================================================

▶ 它是什么？

Embedding 就是把一段文字转换成一个"数字列表"（数学上叫"向量"）。

生活类比：每个城市在地图上有一个经纬度坐标。
  - 北京 ≈ (39.9, 116.4)
  - 上海 ≈ (31.2, 121.5)
  - 两个坐标靠得近的城市，地理位置也近。

Embedding 做的事完全一样，只是把"地理位置"换成了"语义位置"：
  - "猫" → [0.12, -0.33, 0.87, ..., 0.04]  （一个 1536 个数字的列表）
  - "狗" → [0.11, -0.31, 0.85, ..., 0.06]  （和"猫"的坐标很接近！）
  - "笔记本电脑" → [-0.52, 0.71, -0.21, ..., 0.33]  （和"猫"的坐标很远）

"语义相近的文字 → 在向量空间中坐标也相近"，这就是 Embedding 的魔力。

▶ 它解决什么问题？

计算机只懂数字，不懂文字。你没法让计算机判断 "你好" 和 "您好" 有多相似——
除非你把它们都变成数字。Embedding 就是这个"翻译官"：文字入，数字出。

有了数字坐标，就可以用数学公式（余弦相似度）精确计算两段文字的语义距离。

▶ 它怎么工作的？

Embedding 模型（如 OpenAI 的 text-embedding-3-small）是一个训练好的神经网络。
训练过程用了一个巧妙的思路：

  1. 从互联网上抓取海量文本（维基百科、书籍、网页）
  2. 把每段文字中的"相邻句子"当作正例，"随机句子"当作负例
  3. 训练模型：让正例的向量靠得更近，负例的向量离得更远
  4. 训练完成后，模型学会了"把意思相近的文字映射到相近的坐标"

关键参数（你需要知道的）：
  - 维度（dimensions）：向量有多少个数字。OpenAI 默认 1536 维，可以降到 256/512。
    维度越高 → 语义表达能力越强，但存储和计算成本也越高。
  - 模型选择：text-embedding-3-small（便宜，够用）vs text-embedding-3-large（更准，更贵）

▶ 代码长什么样？（概念预览，今天不跑）

```python
from openai import OpenAI
client = OpenAI()

# 一段文字 → 一个 1536 维的数字列表
response = client.embeddings.create(
    model="text-embedding-3-small",
    input="猫咪是一种很可爱的动物"
)

vector = response.data[0].embedding
# vector = [0.0123, -0.0456, 0.0789, ...]  # 1536 个浮点数
# 这个东西就是这段文字的"语义坐标"
```


五、概念 2：Node（节点）——"把书切成卡片"
===========================================

▶ 它是什么？

Node 是 LlamaIndex 的数据原子单位。你可以把它理解成一张"知识卡片"。

生活类比：
  你有一本 500 页的《员工手册》。你不想每次查资料都翻整本书。
  所以你把书拆成一张张卡片：每张卡片抄一段关键内容，标上页码。
  - 一张卡片 = 一个 Node
  - 卡片上的内容 = Node 的 text 属性
  - 卡片标注的页码 = Node 的 metadata（元数据）

▶ 它解决什么问题？

两个问题：

  问题 A：整篇文档塞不进 LLM 上下文
    一篇 PDF 可能 5 万字，GPT-4 上下文 128K tokens（约 9.6 万中文字）。
    一篇就快满了，你还想查 500 篇文档？不可能。必须切碎。

  问题 B：大段文字检索精度差
    如果你把整本书变成一个 Embedding 向量，检索时只知道"这本书相关"，
    但不知道书里哪一段相关。切成小段后，每一段都有自己的向量，
    检索结果能精确到段落级别。

▶ 怎么切？（三种策略对比）

| 策略 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| 固定大小分块 | 每 N 个字符切一刀 | 简单、可控 | 可能在句子中间切断 |
| 语义分块 | 在段落/句子边界切 | 语义完整 | 块大小不均 |
| 递归分块 | 按分隔符优先级递归切 | 平衡两者 | 实现复杂 |

两个关键参数（Day 2 你会亲自动手调）：
  - chunk_size：每块多大（常用 512 或 1024 tokens）
  - chunk_overlap：相邻两块重叠多少（常用 10%-20%）
    重叠是为了防止关键信息正好卡在"刀口"上被切断。

示意图：
  ┌─ 无 overlap ──────────────────┐  ┌─ 有 overlap ────────────────────┐
  │ [AAAAA][BBBBB][CCCCC]         │  │ [AAAAA]                         │
  │                                │  │   [AABBB]  ← 重叠部分保留上下文  │
  │ 如果关键信息在 A 末尾和 B 开头 │  │      [BBCCC]                    │
  │ 切开后就拼不回来了             │  │         [CCCCC]                 │
  └────────────────────────────────┘  └────────────────────────────────┘


六、概念 3：Index（索引）——"给卡片建编号系统"
=================================================

▶ 它是什么？

Index 是一个数据结构，用于快速查找"和某段文字最相似的 Node"。

生活类比：
  图书馆的书架如果没有编号系统，你找一本书要翻遍整个图书馆。
  有了杜威十进制分类法，你直接走到对应的书架就行。
  Index 就是向量数据库里的"编号系统"——把每个 Node 的向量坐标
  按特定方式组织起来，让查找加速。

▶ 它解决什么问题？

如果没有 Index，每次检索要做的事是：
  用户问题的向量 vs 所有文档的向量，逐个算距离。

  1000 个文档？算 1000 次，秒出。
  100 万个文档？算 100 万次，要等很久。
  1 亿个文档？算 1 亿次，没法用了。

Index 通过巧妙的算法（如 HNSW、IVF）提前组织好向量，让检索从
"逐个对比"变成"跳着找"，大幅加速。

▶ 常见索引类型

  - Flat Index（暴力检索）：不做任何优化，逐个对比。精确但慢。
    适用场景：文档量 < 1 万。
  - HNSW（分层可导航小世界图）：把向量组织成多层图结构，
    从上层（粗略）往下层（精确）跳着找。精度高、速度快。
    适用场景：大部分实际项目。
  - IVF（倒排文件索引）：先用聚类把向量分到 N 个桶里，
    查询时只搜最近的一两个桶。速度快但精度略降。

▶ 你现在不需要理解每种索引的内部算法，知道"它存在、它加速检索"就够了。


七、概念 4：Retriever（检索器）——"根据问题找卡片"
=====================================================

▶ 它是什么？

Retriever 是"问答阶段"的第一步：接收用户问题，从 Index 中找出最相关的 Node。

生活类比：
  你在图书馆问管理员："关于 Python 报错处理的书在哪里？"
  管理员去查编号系统 → 锁定 3 本书 → 把书递给你。
  Retriever 就是这个管理员。

▶ 它怎么工作的？

  输入：用户问题 → Embedding 模型 → 问题向量
  过程：问题向量 与 Index 中所有 Node 向量做相似度计算
  输出：Top-K 个最相似的 Node（K 通常是 3-10）

▶ 相似度怎么算？

三种常见算法：

  a) 余弦相似度（Cosine Similarity）——最常用
     计算两个向量之间的夹角余弦值。
     值域 [-1, 1]，1 = 完全同向（语义完全相同），0 = 无关，-1 = 反向。

  b) 欧几里得距离（Euclidean Distance）
     计算两个点在空间中的直线距离。
     越小越相似。但对高维向量效果不如余弦相似度（维度灾难）。

  c) 点积（Dot Product）
     两个向量对应位置相乘再求和。值越大越相似。
     要求向量已归一化（每个向量的长度为 1）。

▶ 检索质量取决于什么？

  1. Embedding 模型好不好 → 语义相近的向量是否真的靠得近
  2. 分块策略是否合理 → Node 的粒度是否匹配问题的粒度
  3. Top-K 选多大 → K 太小可能漏掉关键信息，K 太大引入噪音
  4. 相似度阈值 → 低于某个分数的结果应该丢弃（防止"硬凑"）


八、概念 5：QueryEngine（查询引擎）——"翻完书，写答案"
=========================================================

▶ 它是什么？

QueryEngine 是整个 RAG 流程的"指挥中心"。
它编排了：检索 → 拼接 Prompt → 调用 LLM → 格式化输出 的全过程。

生活类比：
  回到"开卷考试"的类比：
  - Retriever = 翻书找到相关段落
  - QueryEngine = 看完整道题 → 翻到相应段落 → 结合段落内容写出答案 → 标上引用出处

  你不是"翻到哪页抄哪页"，你要用自己的话整理答案，LLM 做的就是这个。

▶ 它怎么工作的？（一次完整的问答调用）

  Step 1: 接收用户问题 "What is RAG?"

  Step 2: 把问题发给 Retriever，拿到 Top-3 相关 Node：
    Node 1: "RAG stands for Retrieval-Augmented Generation..." (score: 0.92)
    Node 2: "The key advantage of RAG is..." (score: 0.87)
    Node 3: "RAG was first introduced by..." (score: 0.81)

  Step 3: 拼接 Prompt 模板：
    """
    你是一个知识助手。请根据以下参考资料回答用户问题。
    如果参考资料中没有相关信息，请如实说"不知道"。

    参考资料：
    [1] RAG stands for Retrieval-Augmented Generation...
    [2] The key advantage of RAG is...
    [3] RAG was first introduced by...

    用户问题：What is RAG?
    请用中文回答，并在答案中标注引用来源（如 [1]、[2]）。
    """

  Step 4: 调用 LLM，传入拼接好的 prompt

  Step 5: 返回 LLM 生成的答案（带引用标注）

▶ 两种查询模式

  - 直接问答（Query）：用户问一句，系统答一句。独立上下文。
  - 对话模式（Chat）：保留对话历史，支持追问。
    例如："那它和 Fine-tuning 有什么区别？"——"它"指代上一轮的 RAG。
    这需要把对话历史也传给 LLM（多轮对话，你 Week 1 Day 3 已经学过了）。

▶ 高级特性（Week 3 Day 4 会用到）

  - Hybrid Search（混合检索）：向量检索 + 关键词检索，两者结果融合。
  - Re-ranking（重排序）：先用简单检索拿 Top-20，再用更精准的模型重排到 Top-5。
  - 引用追溯（Citation）：让 LLM 在答案中标注 [1]、[2]，用户能看到信息出处。


九、五个概念的协作关系——一张总结表
=====================================

| 概念 | 类比 | 输入 | 输出 | 发生在 |
|------|------|------|------|--------|
| Embedding | 翻译官 | 文字 | 向量（数字列表） | 入库 + 问答 |
| Node | 知识卡片 | 文档片段 | 带元数据的文本块 | 入库 |
| Index | 图书馆编号 | 所有 Node 的向量 | 可快速检索的数据结构 | 入库 |
| Retriever | 图书管理员 | 用户问题的向量 | Top-K 相关 Node | 问答 |
| QueryEngine | 考生 | 用户问题 | 带引用的最终答案 | 问答 |


十、今天最重要的 5 个认知
===========================

1. RAG = 开卷考试。LLM 不靠记忆回答，而是先"翻书"再回答。

2. Embedding 是整个 RAG 的基石。把文字变成数字坐标，语义相近的文字坐落在
   相近的坐标上——这是所有检索魔法的基础。

3. 分块（Chunking）是 RAG 系统最容易踩坑的地方。切太大 → 检索不准，
   切太小 → 丢失上下文。chunk_overlap 是你明天要重点实验的参数。

4. 向量检索本质是"最近邻搜索"——在高维空间里找到离问题最近的几个文档向量。
   数学很简单（余弦相似度），工程很复杂（百万/亿级向量怎么加速）。

5. RAG 不能解决所有问题。如果检索到的文档不包含答案，LLM 还是会瞎编。
   所以 Day 4 会学"检索质量评估"和"Re-ranking"，Day 6 会做端到端评测。
