一、先感受"不分块"的痛——为什么需要 Chunking？
================================================

▶ 回到 Day 1 的图书馆类比，加入一个新场景：

  你有一本 500 页的《员工手册》，你想用 RAG 回答员工关于手册的问题。

  如果不分块，你有两个选择：

  选择 A：把整本 500 页的书做成一个 Embedding 向量
    → 检索时找到"这本书相关"，但你不知道是第 3 页还是第 300 页相关
    → Embedding 模型本身也有输入长度限制（text-embedding-3-small 最大 8191 tokens）
    → 500 页约 25 万中文字 ≈ 35 万 tokens，远超限制

  选择 B：把每句话做成一个 Embedding 向量
    → "公司"、"实行"、"弹性"、"工作制"——单句太短，缺少上下文
    → Embedding 模型很难理解孤立的碎片

  分块（Chunking）就是在这两个极端之间找平衡：
    每块不能太大（检索不准）也不能太小（丢失上下文）。

▶ 用一个生活类比理解 chunk_size：

  你把书做成一套"知识卡片"：
    - 卡片太大（一章一张）→ 你知道相关信息在这张卡上，但卡上内容太多，LLM 消化不了
    - 卡片太小（一句一张）→ 信息碎片化，"弹性工作制"的卡片脱离了"考勤制度"的上下文
    - 卡片刚好（一段一张）→ 每张卡片是一段完整的语义单元，检索精准，LLM 也好理解

  今天你会亲自动手调节这个"卡片大小"。


二、文档的"骨架"——先理解文档结构
====================================

在写分块代码之前，先理解你面对的是什么。

▶ 你的 SAMPLE_DOC 是一份 Markdown 格式的《员工手册》，结构如下：

  # 员工手册（2026 版）          ← 一级标题（整本书的名字）
  ## 第一章：入职指引            ← 二级标题（一章）
  ## 第二章：考勤与假期          ← 二级标题
  ## 第三章：报销制度            ← 二级标题
  ## 第四章：IT 与安全           ← 二级标题
  ## 第五章：代码规范            ← 二级标题

  每个 ## 章节内部有若干段落。段落之间有空行分隔。

▶ 分块策略的终极问题是：刀往哪切？

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  ## 第二章：考勤与假期                                    │
  │                                                         │
  │  公司实行弹性工作制，核心工作时间为 10:00-16:00。         │  ← 第 1 段
  │                                                         │
  │  年假政策：入职满 1 年的员工享有 5 天带薪年假...          │  ← 第 2 段
  │                                                         │
  │  病假需提供医院证明，事假每年累计不超过 10 天...          │  ← 第 3 段
  │                                                         │
  └─────────────────────────────────────────────────────────┘

  好的分块：在段落边界切（保留每段的完整性）
  坏的分块：在第 47 个字符处硬切（可能切在句子中间）

  三种策略就是三种"找刀口"的方法——下面逐个讲。


三、三种分块策略——同一个文本，三种切法
========================================

▶ 先看代码骨架（你要理解每一行）：

```python
# 每一块（Chunk）在代码中表示为一个字典：
chunk = {
    "text": "实际文本内容...",
    "start": 120,       # 这个 chunk 在原文中的起始字符位置
    "end": 240,         # 这个 chunk 在原文中的结束字符位置
    "strategy": "..."   # 用哪种策略产生的
}
```

三个字段的作用：
  - text：分块后的文本内容，这是最终要送进 Embedding 模型的
  - start/end：记录这个 chunk 在原文中的位置，方便用户追溯原文
  - strategy：标记来源（后面评估时用）


▶ 策略 A：固定大小分块（Fixed-Size Chunking）

  做法：每 N 个字符切一刀，不管内容在哪断。

  ```python
  def fixed_size_chunk(text, chunk_size=120):
      chunks = []
      for i in range(0, len(text), chunk_size):
          chunk_text = text[i : i + chunk_size]
          if chunk_text.strip():
              chunks.append({"text": chunk_text, "start": i, "end": i + chunk_size})
      return chunks
  ```

  生活类比：用裁纸机切纸，刀的位置是固定的——第 120 张纸一刀，第 240 张一刀。

  优点：实现最简单，chunk 大小均匀，token 消耗可精确预估。
  缺点：可能在句子中间切断，破坏语义完整性。

  示例：
    "年假政策：入职满 1 年的员工享有 5 天带薪年假，满 3 年 10 天，"
    → 如果 chunk_size=50，这一句就会被切成两半，前半在 Chunk A，后半在 Chunk B
    → 检索"年假有多少天"时只命中 Chunk A，看不到"10天"的信息

  适用场景：文档量非常大、对精度要求不高的初筛阶段。


▶ 策略 B：句子边界分块（Sentence-Based Chunking）

  做法：在句号（。）、问号（？）、感叹号（！）、换行（\n）处切，
       保证每个 chunk 都以完整句子结尾。

  ```python
  def sentence_chunk(text, max_chunk_size=200):
      # 先找到所有"句子边界"
      boundaries = []
      for i, ch in enumerate(text):
          if ch in "。！？\n" and i > 0:
              boundaries.append(i + 1)

      # 按边界切，但每个 chunk 不超过 max_chunk_size
      chunks = []
      start = 0
      for boundary in boundaries:
          if boundary - start <= max_chunk_size:
              continue  # 还没超限，继续积累
          # 超限了，切！
          chunks.append({"text": text[start:boundary], ...})
          start = boundary
      return chunks
  ```

  生活类比：裁缝剪布，沿着缝线（句子边界）剪，而不是对着图案中间一刀下去。

  优点：句子完整性好，语义单元完整。
  缺点：chunk 大小不均匀——有些句子很长（200+ 字），有些很短（10 字）。

  适用场景：文档以自然语言为主（文章、手册、报告）。


▶ 策略 C：递归分块（Recursive Chunking）——LangChain 也用这个思路

  做法：按分隔符优先级递归切分。先粗后细，像整理文件柜。

  优先级链：双换行（段落） > 单换行（行） > 句号 > 问号 > 感叹号 > 逗号 > 空格

  ```
  def recursive_chunk(text, chunk_size=150):
      separators = ["\n\n", "\n", "。", "！", "？", "，", " "]

      def _split(t, sep_idx=0):
          if sep_idx >= len(separators):
              # 最后手段：所有分隔符都用不上，固定大小硬切
              return [t[i : i + chunk_size] for i in range(0, len(t), chunk_size)]

          sep = separators[sep_idx]
          if sep not in t:
              return _split(t, sep_idx + 1)  # 这个分隔符不存在，换下一级

          parts = t.split(sep)
          result = []
          for part in parts:
              if len(part) <= chunk_size:
                  result.append(part)  # 这段够小了，收下
              else:
                  result.extend(_split(part, sep_idx + 1))  # 还是太大，继续切
          return result

      return _split(text)
  ```

  生活类比：整理一个塞满文件的柜子。
    先按年份分（\n\n），年份内按月份分（\n），同一个月内按日期分（。）
    如果某一堆文件还是太多，再按更细的标准分——直到每堆都"差不多大"。

  优点：平衡了大小均匀和语义完整。
  缺点：实现复杂一点，有些递归调用需要理解。

  适用场景：大部分 RAG 项目的主力方案。

五、chunk_overlap——"拍照时的重叠区"
=====================================

▶ 它是什么？

  假设 chunk_size=100，overlap=20：
    Chunk 1: 字符 0-100
    Chunk 2: 字符 80-180  ← 前 20 个字和 Chunk 1 重叠
    Chunk 3: 字符 160-260  ← 前 20 个字和 Chunk 2 重叠

  就像拍全景照片：每张照片和相邻照片有重叠部分，拼起来不漏。

▶ 为什么要重叠？

  场景：有一句关键信息是 "员工每年有 15 天带薪年假。"
  如果这一句刚好被切在两个 chunk 的边界上：

  无 overlap：
    Chunk A: "...入职满 5 年 15 天。年假需提前一周..." ← 这句被拆散
    → 用户问"年假多少天"时，两个 chunk 各有一半信息

  有 overlap (20字)：
    Chunk A: "...入职满 5 年 15 天。年假需提前一周向直属..."
    Chunk B: "15 天。年假需提前一周向直属上级申请。病假需..." ← 完整的句子被保留
    → 用户问"年假多少天"时，Chunk B 里有完整信息

▶ Overlap 的参数怎么选？

  chunk_overlap 通常设为 chunk_size 的 10%-20%：
    - chunk_size=200 → overlap=20~40
    - chunk_size=500 → overlap=50~100
    - chunk_size=100 → overlap=10~20

  overlap 太小：起不到"桥接"作用
  overlap 太大：太多重复内容，浪费 Embedding 和存储空间

  这个参数没有"完美值"——不同文档类型需要不同的设置，需要通过评测来选定。


六、完整索引流水线——今天的主角
================================

现在把 Day 1 的理论 + Day 2 的 ChromaDB + 今天的分块串联起来。

▶ 入库阶段（Indexing）的完整流程：

  原始文件 (.txt/.md)
      ↓  Step 1: 读取
  文本字符串
      ↓  Step 2: 分块（今天的核心）
  List[Chunk]  （每个 chunk 约 200 字符）
      ↓  Step 3: Embedding（调用 OpenAI text-embedding-3-small）
  List[Vector] （每 chunk → 1536 维向量）
      ↓  Step 4: 存入 ChromaDB
  Collection  （可被语义搜索）

▶ 问答阶段（Querying）：

  用户问题
      ↓  Step 5: Embedding 问题
  问题向量
      ↓  Step 6: ChromaDB 检索
  Top-K Chunk  （最相关的几个文档片段）
      ↓  Step 7: 拼接 Prompt
  增强后的 Prompt（参考资料 + 用户问题）
      ↓  Step 8: LLM 生成
  最终答案（带引用来源）

Demo 的第四部分会跑通 Step 1-6（入库 + 检索）的完整流程。

八、分块策略选型——一张决策表
===============================

| 场景 | 推荐策略 | chunk_size | overlap | 理由 |
|------|---------|-----------|---------|------|
| 文章/手册/报告 | 递归分块 | 200-500 | 20-50 | 语义完整，大小均匀 |
| 代码文件 | 固定大小 | 500-1000 | 50-100 | 代码每行都是完整语句 |
| FAQ 问答对 | 句子分块 | 100-200 | 0 | 问答对本身就是独立单元 |
| 法律合同 | 递归分块 | 300-500 | 50-80 | 条款不能断裂 |
| 聊天记录 | 固定大小 | 200-400 | 50 | 没有明显段落结构 |

▶ 你现在的阶段：用递归分块 + chunk_size=200 + overlap=30 作为默认配置。
  后面做具体项目时根据文档类型调整。

十、关键参数速查表
===================

| 参数 | 含义 | 你现在的默认值 | 调大的效果 | 调小的效果 |
|------|------|--------------|-----------|-----------|
| chunk_size | 每 chunk 的目标字符数 | 200 | 块更大，上下文更丰富，但检索精度下降 | 块更小，检索更精准，但可能丢失上下文 |
| chunk_overlap | 相邻 chunk 的重叠字符数 | 30 | 信息更保险，但存储和 token 消耗增加 | 存储更省，但可能切断关键信息 |
| top_k (检索) | 检索时返回几个 chunk | 3-5 | 更多上下文给 LLM，但 token 消耗增加 | 更省 token，但可能漏掉关键信息 |

这些参数没有"正确值"，只有"适合你场景的值"。Week 3 Day 5/6 会用评测来选定。


十一、今天的核心认知
====================

1. Chunking 是 RAG 系统中最容易被忽视但影响最大的环节。
   切太大 → 检索不准。切太小 → 丢失上下文。切在句子中间 → 语义支离。

2. 三种策略各有适用场景：
   固定大小：简单可控，适合代码等结构化文本
   句子边界：语义完整，适合自然语言文章
   递归分块：平衡各种需求，是大部分 RAG 项目的主力方案 ← 推荐

3. chunk_overlap 是"保险丝"——防止关键信息恰好被切在边界上。
   10%-20% 的 overlap 是经验起点。

4. 完整索引流水线 = 读取 → 分块 → Embedding → 存储 → 可查询。
   今天你写完这五个步骤的代码，RAG 系统的"入库"阶段就通了。
   "问答"阶段（检索 → 生成）是 Day 4-5 的内容。
